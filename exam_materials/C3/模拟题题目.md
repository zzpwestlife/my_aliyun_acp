# C3 模拟题题目

## 考试说明

- 本卷为 `C3` 中修重做卷，共 `26` 题。
- 单选题 `16` 题，多选题 `10` 题。
- 难度结构：基础题 `8`、中档题 `8`、拔高题 `4`。
- 建议时长：`45` 到 `55` 分钟。
- 建议先独立作答，再对照 `模拟题答案解析.md` 复盘。

## 作答建议

- 先判断题目考的是哪一层边界：工具、规划、协作、记忆、Skill、评测还是 Qwen Code。
- 中档题不要只看“哪个能力更强”，而要看当前约束下哪个方案更稳、更可控。
- 多选题优先排查那些“局部正确，但不能满足题干核心约束”的选项。

## 一、单选题

### 1. 【基础·3.1】某客服 Agent 接到“帮我查一下明天杭州天气并写进日报”的请求时，先输出了一个包含 `tool_name=weather.search` 和参数对象的结构化结果；真正的天气查询和日报写入都由宿主程序完成。这个流程最准确地说明了什么？

- A. 只要模型输出了结构化对象，就等于工具已经执行成功
- B. 工具调用的本质是模型负责决定“调什么、传什么”，执行仍由外部系统完成
- C. 当工具列表足够完整时，模型可以绕过宿主程序直接访问外部 API
- D. 结构化输出的主要作用是让模型记住历史消息，而不是触发外部能力

### 2. 【基础·3.1】某团队发现“创建日程”工具经常失败，复盘后发现失败原因大多是字段缺失、时间格式不统一、布尔值偶尔输出成中文。若只能先做一项改动，哪项最直接？

- A. 给工具调用增加明确的 schema 约束，限定必填字段、类型和格式
- B. 在系统提示词里补充更多业务背景，提醒模型尽量小心
- C. 把工具调用改成先输出自然语言，再由人工二次整理
- D. 让模型每次先解释自己的思路，再给最终参数

### 3. 【中档·3.1】一个企业 Agent 平台准备统一接入 CRM、工单、飞书日历和内部审批系统。团队不希望为每种服务都维护一套独立的调用协议，但又要保留能力发现、参数描述和返回结果的标准化。以下哪种做法更合适？

- A. 针对每个系统分别写长 Prompt，让模型记住它们的接口细节
- B. 把所有接口说明全文塞进系统提示词，依赖模型自由组织调用参数
- C. 把外部能力抽象成统一的工具描述协议，再由宿主按同一规则暴露给模型
- D. 优先做微调，让模型直接背下所有 API，再减少协议层复杂度

### 4. 【中档·3.2】某财务处理任务长期稳定，流程始终是“下载附件 -> OCR 抽取 -> 规则校验 -> 生成报表 -> 通知负责人”。业务方要求每一步都可审计、失败后能知道停在第几步，且不希望 Agent 自己临时改顺序。优先采用哪种组织方式更合理？

- A. 让多个 Agent 共享一个白板，自由协商接下来做什么
- B. 预设固定工作流，或使用显式的 Plan & Execute 逐步推进
- C. 仅依靠长期记忆记录以前怎么做过，让模型本轮自行决定顺序
- D. 只保留一个通用 Agent，让它边想边试，保证灵活性最大

### 5. 【中档·3.2】某代码生成 Agent 在本地静态检查全部通过，但一到集成环境就因为缺少运行参数和真实依赖而报错。团队希望尽快知道问题到底出在“代码逻辑”还是“运行条件”。此时最有效的下一步是？

- A. 让 Agent 不看任何报错信息，再独立反思一轮，避免被噪声干扰
- B. 立刻换更大的模型，优先排除模型能力不足
- C. 先删掉执行环节，只保留静态分析，避免重复失败
- D. 把真实执行日志、退出码和失败步骤回传给 Agent，再让它基于证据修正

### 6. 【中档·3.3】某系统由一个调度 Agent 负责接收任务、拆分子任务、把它们分别交给检索 Agent、写作 Agent 和审校 Agent，并在最后统一汇总输出。这个系统最接近哪种协作模式？

- A. Leader-Worker，由中心角色负责拆解、分配和汇总
- B. handoff，由各 Agent 顺序交接控制权且没有中心调度
- C. Blackboard，由所有 Agent 只围绕共享中间状态并行写作
- D. 长期记忆协作，由角色差异主要体现在记忆召回结果不同

### 7. 【中档·3.3】一个创意策划系统里，没有固定总控 Agent。文案 Agent、设定 Agent、校对 Agent 都围绕同一份“项目白板”持续读写中间结果：有人补充设定，有人重写语气，有人标注冲突点。这个系统更接近哪种模式？

- A. 固定工作流，因为所有参与者都围绕同一任务目标推进
- B. Leader-Worker，因为最终仍然会输出一份统一结果
- C. Blackboard，因为多个 Agent 通过共享中间空间协作迭代
- D. handoff，因为每个 Agent 只会接手上一个 Agent 的最终产物

### 8. 【基础·3.4】某团队把上一轮聊天记录完全不传给模型，结果模型对同一个用户前文提到的项目名、约束条件和已做决定全部“失忆”。这最直接说明当前大模型 API 默认具有什么特性？

- A. 默认具备长期项目记忆，只是需要更长提示词才能激活
- B. 默认是无状态调用，连续对话需要外部显式携带上下文
- C. 默认会自动把历史高频信息固化为长期记忆
- D. 默认依赖缓存机制恢复上轮状态，只是本次缓存失效

### 9. 【中档·3.4】某客服 Agent 要处理一个超长投诉会话。早期消息里包含用户身份、历史订单和已承诺的补偿方案，但上下文窗口已经快满了。团队不希望完全丢掉早期信息，也不想把所有原文无差别写进长期记忆。此时更合适的做法是？

- A. 直接截断到最近几轮，反正最新消息通常最重要
- B. 把所有历史消息都永久写入长期记忆，后面需要时再检索
- C. 每轮都重新让用户概括一次关键信息，避免系统做任何压缩
- D. 用滚动摘要压缩长对话，尽量保留后续决策真正依赖的背景

### 10. 【基础·3.5】某团队每次做代码审查都临时拼接很长 Prompt，不同成员写法差异很大，结果同样的代码在不同人手里得到完全不同的审查深度。若目标是沉淀可复用、可共享、可演进的能力，最适合的做法是？

- A. 把这套审查流程固化为 Skill，明确触发条件、步骤和资源依赖
- B. 保留自由发挥，只要求成员把 Prompt 写得更完整
- C. 每次把上次对话历史一起贴进去，依赖会话延续保持一致性
- D. 先把审查意见存进长期记忆，后续由模型自己归纳成固定流程

### 11. 【中档·3.6】某 Agent 系统上线后整体成功率下降。团队怀疑问题可能出在“检索质量下降”“工具调用参数退化”或“生成阶段引用错证据”中的某一处，但仅看最终回答很难区分。此时优先应该做什么？

- A. 继续扩大端到端评测集，等样本足够多后再猜具体是哪一层出了问题
- B. 直接替换检索模块和生成模型，同时观察是否整体恢复
- C. 为检索、工具调用和生成链路分别设计白盒评测，先拆开定位瓶颈
- D. 先依赖经验判断最可能的模块，再只针对那个模块改动

### 12. 【拔高·综合】某团队要做一个可持续迭代的代码审查 Agent，要求它能读文件、调用搜索、遵循团队审查规范、跨会话记住项目约束，并且每次升级后都能验证质量是否真的变好。以下哪组能力组合最完整？

- A. 更长上下文 + 更强模型 + 主观抽样复查 + 单次 Prompt
- B. 工具调用 + Skill + 持久记忆 + 评测闭环
- C. 会话历史 + 人工复查 + 临时脚本 + 一次性压测
- D. 大模型直答 + 静态规则 + 无状态调用 + 人工经验判断

## 二、多选题

### 13. 【基础·3.1】某 Agent 接入“创建日历事件”工具后，偶尔会输出字段名拼错、时间格式不对或漏掉参会人列表。为了提升工具调用可靠性，以下哪些做法更合理？

- A. 在工具描述或调用约束里明确字段 schema、类型和必填项
- B. 在程序侧对模型产出的参数做校验，而不是直接盲调工具
- C. 校验失败时把具体错误反馈给模型，再尝试生成一版修正参数
- D. 尽量减少程序约束，让模型自由表达，避免束缚它的推理空间

### 14. 【中档·3.2】关于工作流、规划和反馈机制，以下哪些说法更合理？

- A. 步骤固定、可审计要求高的任务，通常更适合固定工作流
- B. 只要任务稍微复杂，Plan & Execute 就一定比固定工作流更优
- C. 当错误只会在真实执行后暴露时，外部反馈通常比纯自我反思更关键
- D. 对高风险任务，把“先拆解再逐步执行并保留中间状态”显式化，通常比完全自由探索更可控

### 15. 【基础·3.3】关于多 Agent 协作模式，以下哪些判断是正确的？

- A. Leader-Worker 更适合存在中心调度和明确任务拆解的场景
- B. Blackboard 更适合多个 Agent 围绕共享中间空间持续共创
- C. handoff 的核心是始终由同一个总控角色并行调度所有参与者
- D. handoff 适合阶段性交接较清晰的串行协作流程

### 16. 【中档·3.4】关于 Agent 的记忆管理，以下哪些做法是合理的？

- A. 用上下文截断控制短期窗口长度，避免无关历史持续占位
- B. 用滚动摘要压缩长对话，同时保留后续决策依赖的关键背景
- C. 对真正跨会话复用的信息建立长期记忆，并通过检索召回
- D. 只要向量库够大，就应该把所有对话原文永久无差别写入长期记忆

### 17. 【中档·3.5】关于 Skill 的设计与演进，以下哪些说法更合理？

- A. 只要某个任务足够高频，就应该立刻固化为 Skill，不必先澄清触发边界
- B. Skill 的触发条件需要尽量清楚，否则容易误触发或漏触发
- C. Skill 放进版本控制后，更利于团队协作、回滚和持续改进
- D. 一个可复用的 Skill 应尽量说明输入输出约束、依赖资源和执行步骤

### 18. 【中档·3.6】关于评测驱动 Agent 开发，以下哪些说法更合理？

- A. 不能只凭“感觉变好了”判断升级是否真的有效
- B. 端到端评测能帮助判断最终输出对用户是否有价值
- C. 只要基线版本通过一次完整评测，后续就可以不再维护回归集
- D. 反复失败的典型样例应沉淀为长期回归集，避免相同退化反复出现

### 19. 【基础·3.7】关于 Qwen Code 的工作方式，以下哪些说法更合理？

- A. `Plan Mode` 更适合先分析方案、厘清改动范围，再决定是否动代码
- B. `/compress` 可以在长任务中压缩上下文，减少历史噪声占位
- C. `/clear` 适合任务切换时主动清理上下文边界，避免不同任务互相污染
- D. EPCC 的 `Explore` 阶段应优先快速改动，理解现状可以放到后面补

### 20. 【拔高·综合】某团队希望打造一个能持续优化的 Agent 开发流程，要求既能接入外部能力，又能沉淀团队规范、记住长期约束，并持续验证系统是否真正变好。以下哪些设计更合理？

- A. 用工具调用或 MCP 连接文件、搜索、执行等外部能力
- B. 把所有团队规范都堆进一次性的系统提示词里，后续不再维护成 Skill
- C. 用持久记忆承载跨会话项目约束，而不是每次重新手工贴背景
- D. 用端到端评测配合白盒评测，让升级结果既能看最终效果也能定位原因

## 三、补充单选题

### 21. 【中档·3.1】某 Agent 接入“创建工单”工具后，偶尔因为网络抖动而超时重试，结果同一个请求被系统创建出两张重复工单。排查发现参数大多是正确的。若只能优先补一项机制，最应该先做什么？

- A. 把提示词写得更保守，让模型生成更谨慎的语言描述
- B. 取消所有自动重试，保证绝不发生重复写入
- C. 让模型每次调用前先自然语言解释一次为什么要创建工单
- D. 在写操作边界增加幂等键或请求去重机制，避免同一意图被重复执行

### 22. 【中档·3.2】以下哪类任务最不适合优先交给 Agent 做完全自主规划，而更适合预设固定工作流？

- A. 方案调研，需要比较多条候选技术路线
- B. 报销审批流，步骤固定且错误执行会带来合规风险
- C. 创意文案改写，需要尝试多种表达风格
- D. 头脑风暴，希望尽量探索多个开放方向

### 23. 【中档·3.3】某系统由三个专业 Agent 顺序协作：需求澄清 Agent 先完成上下文整理，再把结论交给实现 Agent；实现 Agent 处理完后，再把结果移交给测试 Agent。整个流程没有一个中心角色持续拆解和调度全部参与者。这个模式更接近？

- A. handoff
- B. Leader-Worker
- C. Blackboard
- D. 长期记忆检索

### 24. 【中档·3.4】某团队把所有对话原文都写入长期记忆，结果记忆库越来越大，但召回出的内容经常无关、过时，甚至和当前项目阶段冲突。更合理的改进是？

- A. 继续扩大向量库容量，优先解决“存不下”的问题
- B. 完全放弃长期记忆，只靠单次上下文硬撑
- C. 只写入稳定、高价值、跨会话可复用的信息，并为写入增加筛选规则
- D. 每次新任务开始前清空全部长期记忆，避免召回噪声

## 四、补充多选题

### 25. 【中档·3.5】关于 Skill 的触发边界与版本演进，以下哪些说法更合理？

- A. 高频、稳定、可复用的流程更适合沉淀为 Skill
- B. Skill 的触发条件越模糊越好，这样能覆盖更多场景
- C. Skill 纳入版本控制有利于回滚、对比和团队协作
- D. Skill 的输入输出约束、触发条件和资源依赖应尽量写清楚
