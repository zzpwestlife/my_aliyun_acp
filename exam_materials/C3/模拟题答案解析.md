## 一、单选题答案解析

### 1. 【基础·3.1】某 Agent 系统日志显示如下循环：模型输出 `{"tool": "search", "query": "xxx"}` → 系统执行搜索 → 结果返回给模型 → 模型再次输出调用意图。这体现了工具调用的核心机制是？

- A. 模型直接执行工具函数  
- B. 模型输出结构化意图，由外部程序执行  
- C. 每次调用都执行全部可用工具  
- D. 通过提高 temperature 增强工具选择灵活性  

标准答案：B

解析：日志显示的是典型的“决策 - 执行分离”模式。模型只负责输出结构化的调用意图，如 JSON 格式的 `tool` 与 `query`，真正的搜索执行由外部程序完成。这是 Function Calling 机制的核心设计，而非模型直接执行工具、全量调用工具或通过 `temperature` 调节工具选择。

知识点：3.1 工具调用机制中的结构化输出与外部执行分离

考纲映射：智能体与应用 -> 理解智能体运行机制

### 2. 【基础·3.1】某团队对比两种Agent设计：方案A让模型先输出`Thought: 我需要查天气`，再输出`Action: get_weather`；方案B让模型直接输出`Action: get_weather`。在复杂多步任务中，方案A通常表现更稳定，这体现了？

- A. 方案A延迟更低  
- B. 显式推理链有助于多步决策  
- C. 方案B的工具调用格式更规范  
- D. 两种方案本质无差异  

标准答案：B

解析：方案 A 是 ReAct 模式的典型体现，先显式输出推理过程，再执行行动。这种“思考 - 行动”分离结构让模型在多步任务中能追踪自己的推理链条，逐步逼近目标，比直接输出行动更稳定可控。

知识点：3.1 ReAct 框架中的思考 - 行动 - 观察循环

考纲映射：智能体与应用 -> 理解智能体运行机制

### 3. 【基础·3.1】某创业公司开发Agent平台，希望接入第三方天气、地图、日历等工具，但不想为每家的API格式单独写适配代码。他们最需要的技术是？

- A. 对模型进行微调以适配各API  
- B. 使用MCP标准化接入层  
- C. 为每个API写独立提示词模板  
- D. 提高模型的temperature参数  

标准答案：B

解析：题目描述的核心痛点是多工具 API 格式不统一、重复适配成本高。MCP 的设计目标正是提供标准化的工具接入层，工具提供方按 MCP 规范暴露能力，Agent 平台统一按 MCP 消费，实现解耦。

知识点：3.1 MCP 标准化工具接入协议

考纲映射：智能体与应用 -> 理解智能体运行机制

### 4. 【基础·3.2】某Agent执行代码生成任务后，日志显示代码通过了静态检查但运行时报错。按课程推荐的反思策略，此时应优先采用？

- A. 让Agent再次自我检查代码逻辑  
- B. 捕获真实报错信息反馈给Agent  
- C. 直接终止任务并标记失败

标准答案：B

解析：静态检查通过但运行失败，说明问题出在运行时行为。此时最有效的反馈是真实的执行信号，例如报错信息和堆栈追踪。这属于外部反馈，能让 Agent 基于事实调整后续行动。

知识点：3.2 反思机制中的外部反馈

### 5. 【中档·3.2】某RPA团队需要自动化处理每月固定格式的财务报表：下载→解析→校验→生成报告。该任务步骤稳定、输入输出格式明确。最适合采用的Agent组织方式是？

- A. 让Agent完全自主规划每一步  
- B. 预设固定工作流或Plan & Execute  
- C. 使用Blackboard模式让多个Agent共创  
- D. 依赖长上下文记忆而不预设流程  

标准答案：B

解析：题目给出的任务步骤稳定、输入输出格式明确。这类任务更适合固定工作流或 Plan & Execute 模式，先由上层拆解步骤，再按序执行，比完全自主规划更可控，也比 Blackboard 共创更高效。

知识点：3.2 固定工作流与自主规划的适用边界

考纲映射：智能体与应用 -> 构造复杂工作流和多智能体应用

### 6. 【中档·3.3】某系统架构如下：顶层Agent接收用户查询后拆解子任务，分发给3个专门Agent分别处理检索、生成、校验，最后顶层Agent汇总输出。这体现了？

- A. Leader-Worker分层规划模式  
- B. Blackboard共创协作模式  
- C. 单一Agent多轮对话  
- D. 外部API网关模式  

标准答案：A

解析：架构的核心特征是“顶层统筹 + 分层拆解 + 专门执行 + 汇总返回”，这是典型的 Leader - Worker 模式。顶层 Agent 负责协调、拆解与汇总，子 Agent 负责执行各自的子任务。

知识点：3.3 多 Agent 协作中的 Leader - Worker 模式

考纲映射：智能体与应用 -> 构造复杂工作流和多智能体应用

### 7. 【中档·3.3】某创意写作团队开发了多个专家Agent（情节设计、角色塑造、文风润色），它们围绕一个共享的"故事板"持续读写、互相激发想法、迭代优化。这最接近？

- A. Leader-Worker分层规划  
- B. Blackboard共创协作模式  
- C. 固定工作流Pipeline

解析：题目描述的关键要素是多专家 Agent、共享空间、持续读写和相互激发，这正是 Blackboard 共创模式的典型特征。它强调多个角色围绕共享白板协同，而不是由单一 Leader 统一调度。

知识点：3.3 Blackboard 共创模式与共享白板机制

### 8. 【基础·3.4】某开发者在连续调用大模型API时，发现每次新请求如果不携带上一轮对话内容，模型就无法理解之前的上下文。这说明大模型具有？

- A. 无状态特性  
- B. 有状态特性  
- C. 自学习特性  
- D. 离线推理特性

### 9. 【中档·3.4】某客服Agent需要处理长达数小时的复杂投诉，早期对话包含关键背景信息（用户身份、历史订单），但后期对话窗口面临长度限制。相比直接丢弃早期内容，滚动摘要策略的优势是？

- A. 完全避免额外模型调用  
- B. 在压缩长度同时保留核心信息  
- C. 一定比向量召回更准确

标准答案：B

知识点：3.4 滚动摘要与固定窗口截断

考纲映射：智能体与应用 -> 理解智能体运行机制

### 11. 【中档·3.6】某Agent系统的最终回答质量下降，开发团队怀疑是某个中间组件（检索模块或生成模块）出了问题，但不确定具体是哪一个。他们应该优先采用？

### 13. 【基础·3.1】某Agent接入天气查询工具时，发现模型有时输出非JSON格式导致解析失败。为提升可靠性，应采用的机制包括？

- A. 用JSON Schema约束输出结构  
- B. 对模型输出做程序校验  
- C. 校验失败时将错误信息反馈给模型重试  
- D. 完全依赖模型自由输出自然语言  

标准答案：A、B、C

解析：可靠工具调用的关键是“结构化输出 + 程序校验 + 失败重试”闭环。用 JSON Schema 约束结构、用程序验证合法性，并在失败时把错误信息反馈给模型重试，三者共同构成可靠链路。

知识点：3.1 工具调用中的结构化输出、校验与重试

### 14. 【中档·3.2】关于反思与工作流选择，以下说法合理的是？

- C. Plan & Execute强调先生成计划再按步骤执行

### 15. 【基础·3.3】关于多Agent协作模式，以下说法正确的是？

### 16. 【中档·3.4】关于Agent的记忆管理，以下哪些做法合理？

- A. 用上下文截断控制窗口长度

标准答案：A、B、C、D
