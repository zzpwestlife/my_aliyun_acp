## 一、单选题答案解析

### 1. 【基础·3.1】某 Agent 在回答“帮我查明天杭州天气”时，先输出一个结构化对象表示要调用天气工具，再由外部程序真正执行查询。这说明工具调用的核心机制是？

标准答案：B

解析：工具调用的核心不是模型自己执行工具，而是模型先生成结构化调用意图，由外部程序解析后实际执行。这种“模型决策，程序执行”的分工是 Function Calling 的基础。

错项分析：A 把模型当成执行器，混淆了边界；C 夸大了自动选择能力；D 把工具调用误解为采样参数问题。

知识点：工具调用中的结构化输出与外部执行边界

考纲映射：Agent 运行机制与工具调用 -> 运行闭环 / 外部执行边界

### 2. 【基础·3.1】某团队希望降低工具调用失败率，要求模型输出稳定的参数结构，便于程序解析。以下哪项措施最直接？

标准答案：A

解析：如果目标是让程序稳定解析，最直接的手段就是对输出结构做约束，例如限制字段名、字段类型和层次结构。这样可以显著降低“模型想表达得通顺但程序读不懂”的问题。

错项分析：B 只增加文字，不直接保证结构；C 会放大解析风险；D 与提高可靠性相反。

知识点：结构化输出与输出约束

考纲映射：Agent 运行机制与工具调用 -> 结构化输出 / 工具调用可靠性

### 3. 【中档·3.1】某平台接入天气、地图、日历等多个第三方能力，但不希望为每个服务都单独写一套接入协议。此时最需要的能力是？

标准答案：B

解析：题干的关键痛点是“多工具接入协议不统一”。最合适的解决方案是使用统一的工具接入协议，把不同工具包装成同一套暴露方式，降低适配成本。

错项分析：A 微调不解决接口标准问题；C 只会让接入更加碎片化；D 把工具能力错误地挪进了提示词。

知识点：MCP 与统一工具接入

考纲映射：Agent 运行机制与工具调用 -> 工具接入方案选择

### 4. 【中档·3.2】某任务流程长期稳定，始终是“下载数据 -> 清洗 -> 汇总 -> 发邮件”。如果目标是降低不可控性并保证步骤顺序清晰，优先选择哪种组织方式？

标准答案：B

解析：任务步骤固定、可重复、顺序明确时，固定工作流或 Plan & Execute 更适合，因为它们能把步骤提前拆清楚，减少自主探索带来的漂移和不确定性。

错项分析：A 适合更开放的问题；C 更适合共创类任务；D 记忆可以辅助，但不能替代流程定义。

知识点：固定工作流与 Plan & Execute 的适用边界

考纲映射：工作流、规划与反思 -> 固定工作流 / Plan & Execute

### 5. 【中档·3.2】某 Agent 生成代码后静态检查通过，但真实运行报错。为了让 Agent 更快定位问题，最有效的下一步是？

标准答案：B

解析：当问题发生在运行时，最有价值的反馈不是再猜一次，而是真实的执行结果和报错信息。外部反馈能把 Agent 的修正建立在事实之上，而不是建立在模型再次臆测之上。

错项分析：A 只有内部反思，缺少关键证据；C 直接终止会丢掉修正机会；D 会把问题从“如何修复”错误转移成“是否使用工具”。

知识点：外部反馈与运行时反思

考纲映射：工作流、规划与反思 -> 自我反馈与外部反馈

### 6. 【中档·3.3】某系统由一个总控 Agent 接收请求、拆分任务，再分发给检索 Agent、生成 Agent 和校验 Agent，最后总控 Agent 汇总结果。这最符合哪种协作模式？

标准答案：B

解析：题干明显呈现出“一个总控 + 多个专门执行者”的结构，这就是典型的 Leader-Worker 模式。总控负责拆解、调度和汇总，子 Agent 负责专门执行。

错项分析：A 共享白板并非题干重点；C 缺少多角色；D 只是能力接入方式，不是协作模式。

知识点：Leader-Worker 分层协作

考纲映射：多 Agent 协作模式 -> 角色分工与调度逻辑

### 7. 【中档·3.3】几个创意写作 Agent 围绕同一份“故事板”持续补充设定、改写情节、统一风格，没有固定的上下级，而是共享同一中间空间持续迭代。这更接近？

标准答案：B

解析：题干强调的是“共享空间”“持续读写”“共同迭代”，这正是 Blackboard 的核心特征。它更强调共享中间状态，而不是中心化调度。

错项分析：A 要有明显上下级；C 更强调固定顺序；D 只涉及单 Agent。

知识点：Blackboard 共创协作

考纲映射：多 Agent 协作模式 -> Blackboard 共享空间机制

### 8. 【基础·3.4】开发者发现：如果新请求里不重新带入历史消息，模型就像“忘记了”上一轮对话。这最直接说明大模型具有？

标准答案：B

解析：大模型 API 调用通常是无状态的，上一轮历史不会自动保留。系统若想维持连续对话，必须由外部显式传回必要上下文。

错项分析：A 与题干相反；C 容易把统计学习误当成会话记忆；D 与问题无关。

知识点：大模型无状态特性

考纲映射：记忆管理 -> 无状态特性与上下文管理

### 9. 【中档·3.4】某客服 Agent 处理超长投诉时，早期对话里有用户身份和历史订单等关键背景。和直接丢弃早期历史相比，滚动摘要更合适的原因是？

标准答案：B

解析：滚动摘要的价值在于在受限窗口里压缩长度，同时尽量保留任务继续推进所需的核心背景。这是“全保留”和“全丢弃”之间的折中方案。

错项分析：A 不成立，摘要往往要额外处理；C 过于绝对；D 摘要只能覆盖部分需求，不能替代长期记忆体系。

知识点：滚动摘要与上下文压缩

考纲映射：记忆管理 -> 截断与滚动摘要

### 10. 【基础·3.5】某团队每次做代码审查都要临时拼接很长 Prompt，且不同成员风格不一致。为了沉淀稳定能力，最适合的做法是？

标准答案：A

解析：题干要解决的是“重复”“不一致”“难复用”。把这类稳定流程沉淀为 Skill，可以统一触发条件、流程组织和标准，显著提升团队复用性。

错项分析：B 只是继续堆 Prompt；C 只能保留短期痕迹；D 仍然没有形成可复用能力单元。

知识点：从 Prompt 到 Skill 的能力沉淀

考纲映射：Skill 设计与沉淀 -> Prompt 到 Skill 的演进逻辑

### 11. 【中档·3.6】某 Agent 系统最终结果变差，团队怀疑问题出在中间组件，但仅看最终输出无法确定是检索模块还是生成模块导致。此时优先应该做什么？

标准答案：B

解析：如果目标是定位具体瓶颈，只看最终输出是不够的。更有效的方法是针对中间组件建立白盒评测，把问题拆到模块层面逐步定位。

错项分析：A 只看到结果看不到原因；C 代价高且仍不一定知道根因；D 不具备可验证性。

知识点：白盒评测与模块定位

考纲映射：评测驱动开发 -> 白盒评测定位瓶颈

### 12. 【拔高·综合】某团队要构建一个可持续迭代的代码审查 Agent，要求它能读文件、调用搜索、遵循团队规范、跨会话记住项目约束，并持续评估质量。以下哪组能力组合最完整？

标准答案：A

解析：这道题考的是系统完整性。读文件和搜索需要工具调用，遵循团队规范需要 Skill，跨会话约束需要持久记忆，持续评估质量需要评测闭环。四者合起来才是完整系统，而不是“会答题”而已。

错项分析：B 缺少工具、长期记忆和可验证评测；C 更像零散补丁拼接；D 完全缺少持续优化能力。

知识点：工具、Skill、记忆与评测的组合设计

考纲映射：综合能力 -> 多能力组合与工程闭环

## 二、多选题答案解析

### 13. 【基础·3.1】某 Agent 接入工具时，模型偶尔输出了不符合预期的参数结构。为了提升工具调用可靠性，以下哪些做法更合理？

标准答案：ABC

解析：可靠工具调用通常由三部分构成：先限制结构，再做程序校验，失败时把反馈回传给模型重试。三者叠加，才能形成真正稳定的调用链路。

错项分析：D 完全放弃结构化解析，会让程序层几乎无法稳定消费输出。

知识点：工具调用可靠性设计

考纲映射：Agent 运行机制与工具调用 -> 结构化输出 / 校验 / 重试

### 14. 【中档·3.2】关于工作流与反思机制，以下哪些说法更合理？

标准答案：ABC

解析：固定流程适合稳定任务，Plan & Execute 适合先拆解再执行，外部反馈则更适合运行时错误定位。这三项分别对应“组织方式”“执行方式”“修正方式”的合理选择。

错项分析：D 把所有任务都交给自主规划，忽视了任务稳定性和可控性的差异。

知识点：工作流选择与反思策略

考纲映射：工作流、规划与反思 -> 固定工作流 / Plan & Execute / 外部反馈

### 15. 【基础·3.3】关于多 Agent 协作模式，以下哪些判断是正确的？

标准答案：ABC

解析：Leader-Worker 适合层级拆解，Blackboard 适合开放共创，handoff 也是常见协作组织方式。它们共同构成了多 Agent 系统中不同的协作范式。

错项分析：D 不成立，多 Agent 通常会带来额外协调成本和 token 成本。

知识点：多 Agent 协作模式与成本

考纲映射：多 Agent 协作模式 -> 协作方式与适用边界

### 16. 【中档·3.4】关于 Agent 的记忆管理，以下哪些做法是合理的？

标准答案：ABC

解析：上下文截断、滚动摘要和向量召回分别服务于短期控制、历史压缩和长期记忆检索，是常见且合理的组合策略。

错项分析：D 把所有对话一股脑永久写入长期记忆会制造大量噪声，反而削弱检索质量，不属于高质量记忆管理。

知识点：短期记忆与长期记忆组合策略

考纲映射：记忆管理 -> 截断 / 摘要 / 长期记忆检索

### 17. 【中档·3.5】关于 Skill 的设计与演进，以下哪些说法更合理？

标准答案：ABC

解析：Skill 的核心价值就是把反复出现的高价值流程沉淀下来，并以更稳定的触发条件、资源组织和版本化方式持续演进。

错项分析：D 过度绝对。Skill 很重要，但并不意味着所有 Prompt 都应被消灭，探索性任务仍可能依赖 Prompt。

知识点：Skill 的沉淀、组织与版本化

考纲映射：Skill 设计与沉淀 -> 设计质量 / 版本控制

### 18. 【中档·3.6】关于评测驱动 Agent 开发，以下哪些说法更合理？

标准答案：ABC

解析：评测驱动开发强调不能只凭感觉判断；端到端评测关注用户价值；白盒评测帮助定位内部瓶颈。三者共同构成更完整的工程判断框架。

错项分析：D 错在把评测当成一次性动作。真正的评测驱动强调持续循环，而不是上线前一次检查。

知识点：端到端评测、白盒评测与持续迭代

考纲映射：评测驱动开发 -> 端到端 / 白盒 / 闭环意识

### 19. 【基础·3.7】关于 Qwen Code 的工作方式，以下哪些说法更合理？

标准答案：ABC

解析：`Plan Mode` 先分析方案，`/compress` 用于压缩上下文，`/clear` 用于任务切换时清理边界，这三项都符合 Qwen Code 的工作方式。

错项分析：D 把 EPCC 的 `Explore` 反过来了。`Explore` 的重点是先理解现状，再进入修改，而不是先改再补理解。

知识点：Qwen Code 的模式与上下文管理

考纲映射：Qwen Code 实践 -> Plan Mode / 上下文管理 / EPCC

### 20. 【拔高·综合】某团队希望打造一个能持续优化的 Agent 开发流程，要求既能接外部能力、沉淀团队规范，又能记住长期约束并持续迭代。以下哪些设计更合理？

标准答案：ABCD

解析：这是一道综合能力题。外部能力接入、团队规范沉淀、长期约束保留，以及端到端加白盒评测闭环，分别对应一个 Agent 系统真正走向工程化所需要的关键支柱，缺任何一项都会影响系统的长期演进能力。

错项分析：本题四项都是合理设计，关键不在排除，而在理解它们分别对应“能力接入、规范复用、长期记忆、持续优化”四个不同维度。

知识点：Agent 系统工程化能力组合

考纲映射：综合能力 -> 工具 / Skill / 记忆 / 评测闭环

## 三、补充单选题答案解析

### 21. 【中档·3.1】某 Agent 接入“创建工单”工具后，偶尔会因为超时重试而把同一请求创建出两张重复工单。参数本身大多是正确的。若只能先补一项机制，最应该优先做什么？

标准答案：B

解析：这道题考的是“写操作工具”的幂等治理，而不是参数格式正确性。既然参数本身大多无误，真正的问题在于重试时缺少去重机制。给请求增加幂等键或服务端去重，才能避免同一意图被执行两次。

错项分析：A 与重复执行治理无关；C 只是语言层提醒，不能替代系统控制；D 虽然能减少重复，但会让瞬时失败失去恢复能力，不是更稳妥的工程方案。

知识点：写操作工具的幂等与重试治理

考纲映射：Agent 运行机制与工具调用 -> 写操作安全性 / 幂等控制

### 22. 【中档·3.2】以下哪类任务最不适合优先交给 Agent 做完全自主规划，而更适合预设固定工作流？

标准答案：B

解析：报销审批流步骤固定、错误成本高、合规要求明确，这类任务更适合预设流程和显式校验，而不是让 Agent 自由探索路径。自主规划更适合目标开放、路径不确定的问题。

错项分析：A、C、D 都更偏开放式探索或创作场景，更能发挥自主规划价值。

知识点：固定工作流与自主规划边界

考纲映射：工作流、规划与反思 -> 任务适配性 / 可控性边界

### 23. 【中档·3.3】某系统包含多个专业 Agent：需求澄清 Agent 完成后，会把上下文和当前结论交给实现 Agent；实现 Agent 完成后，再移交给测试 Agent。整个过程中没有统一总控角色持续调度。这个模式更接近？

标准答案：B

解析：题干强调的是“一个角色做完后把控制权交给下一个角色”，而不是“总控持续拆解并调度多个执行者”。这正是 handoff 的典型特征。

错项分析：A 需要明显的中心调度者；C 强调共享中间空间；D 只是信息存取机制，不是协作模式。

知识点：handoff 与 Leader-Worker 的区别

考纲映射：多 Agent 协作模式 -> handoff / Leader-Worker 边界

### 24. 【中档·3.4】某团队把所有对话内容都写入长期记忆，结果记忆库越来越大，但召回出来的信息经常无关甚至过时。更合理的改进是？

标准答案：B

解析：长期记忆不是“存得越多越好”，而是“存得越准越好”。只有稳定、高价值、跨会话可复用的信息才值得进入长期记忆，同时需要为写入设置规则，避免把噪声和一次性信息持续沉积进去。

错项分析：A 会继续扩大噪声；C 只靠上下文无法覆盖跨会话场景；D 又走向另一个极端，失去长期记忆价值。

知识点：长期记忆写入策略与噪声控制

考纲映射：记忆管理 -> 长期记忆治理 / 写入策略

## 四、补充多选题答案解析

### 25. 【中档·3.5】关于 Skill 的触发边界与版本演进，以下哪些说法更合理？

标准答案：ACD

解析：Skill 适合沉淀高频、稳定、可复用的流程；同时，高质量 Skill 需要清晰的触发条件、输入输出约束和依赖资源。纳入版本控制后，团队也更容易做回滚、对比和协作演进。

错项分析：B 错在把“模糊”误当成“通用”。触发条件越模糊，越容易误触发或漏触发。

### 26. 【拔高·3.6】关于评测闭环如何指导下一轮 Agent 优化，以下哪些做法更合理？

标准答案：ABC
