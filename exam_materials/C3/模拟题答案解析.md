## 一、单选题答案解析

### 1. 【基础·3.1】某客服 Agent 接到“帮我查一下明天杭州天气并写进日报”的请求时，先输出了一个包含 `tool_name=weather.search` 和参数对象的结构化结果；真正的天气查询和日报写入都由宿主程序完成。这个流程最准确地说明了什么？

标准答案：B

解析：这道题考的是工具调用的责任边界。模型的职责是表达“想调哪个工具、传什么参数”，真正的副作用执行仍然由宿主程序或外部系统完成。结构化输出只是调用意图，不等于工具已经被执行。

错项分析：A 把“生成调用计划”和“执行工具”混为一谈；C 高估了模型权限，模型本身并不会直接访问外部 API；D 把结构化输出错理解成记忆机制。

知识点：工具调用中的“模型决策、宿主执行”边界

考纲映射：Agent 运行机制与工具调用 -> Function Calling / 外部执行边界

### 2. 【基础·3.1】某团队发现“创建日程”工具经常失败，复盘后发现失败原因大多是字段缺失、时间格式不统一、布尔值偶尔输出成中文。若只能先做一项改动，哪项最直接？

标准答案：A

解析：题干里的问题都是结构层问题，不是业务理解问题。最直接的修复是给工具调用施加明确的 schema 约束，把字段、类型、格式和必填项显式化，优先提升程序可解析性和调用稳定性。

错项分析：B 可以补充背景，但不直接解决字段格式错乱；C 把自动化问题改回人工整理，成本过高；D 会增加输出长度，却不保证结构更稳定。

知识点：结构化输出约束与工具调用可靠性

考纲映射：Agent 运行机制与工具调用 -> 参数 schema / 稳定输出

### 3. 【中档·3.1】一个企业 Agent 平台准备统一接入 CRM、工单、飞书日历和内部审批系统。团队不希望为每种服务都维护一套独立的调用协议，但又要保留能力发现、参数描述和返回结果的标准化。以下哪种做法更合适？

标准答案：C

解析：题干的核心约束是“多系统统一接入”和“标准化描述能力”。因此更合理的做法是把不同外部能力抽象成统一协议，再由宿主按同一规则暴露给模型。这样既保留统一接入，又降低后续接新系统的适配成本。

错项分析：A 和 B 都把协议问题转嫁到 Prompt 层，后续维护会越来越碎；D 微调可以提升熟悉度，但不能替代统一接入协议。

知识点：统一工具协议与能力暴露标准化

考纲映射：Agent 运行机制与工具调用 -> MCP / 统一工具接入

### 4. 【中档·3.2】某财务处理任务长期稳定，流程始终是“下载附件 -> OCR 抽取 -> 规则校验 -> 生成报表 -> 通知负责人”。业务方要求每一步都可审计、失败后能知道停在第几步，且不希望 Agent 自己临时改顺序。优先采用哪种组织方式更合理？

标准答案：B

解析：这类任务步骤固定、风险明确、强调审计和可控性，最适合预设固定工作流，或使用显式计划后逐步执行的方式。核心不是“能不能做出来”，而是“步骤顺序和中间状态是否可控、可追踪”。

错项分析：A 更适合开放共创；C 可以辅助记忆，但不能代替流程定义；D 灵活性更高，但与题干的可审计、不可改顺序要求冲突。

知识点：固定工作流与显式计划执行

考纲映射：工作流、规划与反思 -> 固定工作流 / Plan & Execute

### 5. 【中档·3.2】某代码生成 Agent 在本地静态检查全部通过，但一到集成环境就因为缺少运行参数和真实依赖而报错。团队希望尽快知道问题到底出在“代码逻辑”还是“运行条件”。此时最有效的下一步是？

标准答案：D

解析：当故障只会在真实执行环境暴露时，最有价值的不是继续猜，而是把真实执行日志、退出码和失败步骤回灌给 Agent。这样才能让它基于运行证据定位到底是逻辑缺陷、环境缺失还是参数准备问题。

错项分析：A 纯自我反思缺少事实依据；B 换模型不能直接解决证据缺失；C 删除执行环节只会把运行问题继续遮住。

知识点：外部反馈优先于纯自我反思

考纲映射：工作流、规划与反思 -> 外部反馈 / 运行时修正

### 6. 【中档·3.3】某系统由一个调度 Agent 负责接收任务、拆分子任务、把它们分别交给检索 Agent、写作 Agent 和审校 Agent，并在最后统一汇总输出。这个系统最接近哪种协作模式？

标准答案：A

解析：题干明确存在一个中心角色负责拆解、分配和汇总，这是典型的 Leader-Worker。判断关键不是“有多个 Agent”，而是是否存在持续负责调度的中心角色。

错项分析：B handoff 更强调阶段性交权；C Blackboard 更强调共享中间空间；D 长期记忆不是协作组织模式。

知识点：Leader-Worker 的调度边界

考纲映射：多 Agent 协作模式 -> Leader-Worker

### 7. 【中档·3.3】一个创意策划系统里，没有固定总控 Agent。文案 Agent、设定 Agent、校对 Agent 都围绕同一份“项目白板”持续读写中间结果：有人补充设定，有人重写语气，有人标注冲突点。这个系统更接近哪种模式？

标准答案：C

解析：题干的关键是“没有总控”和“持续读写共享白板”。这正是 Blackboard 协作的典型特征，多个 Agent 围绕同一中间状态持续迭代，而不是由中心角色逐个分派任务。

错项分析：A 有共同目标不等于固定工作流；B 缺少中心调度者；D handoff 更强调一个阶段结束后整体移交给下一个阶段。

知识点：Blackboard 共创模式

考纲映射：多 Agent 协作模式 -> Blackboard

### 8. 【基础·3.4】某团队把上一轮聊天记录完全不传给模型，结果模型对同一个用户前文提到的项目名、约束条件和已做决定全部“失忆”。这最直接说明当前大模型 API 默认具有什么特性？

标准答案：B

解析：这道题考的是 API 层的默认行为。当前大模型调用通常是无状态的，如果不显式带回上下文，模型并不会“自动记得”上一轮发生了什么。

错项分析：A 和 C 都把模型误解成自带跨轮记忆系统；D 与题干无关，缓存也不是默认会话记忆机制。

知识点：无状态调用与上下文显式传递

考纲映射：记忆管理 -> 无状态特性

### 9. 【中档·3.4】某客服 Agent 要处理一个超长投诉会话。早期消息里包含用户身份、历史订单和已承诺的补偿方案，但上下文窗口已经快满了。团队不希望完全丢掉早期信息，也不想把所有原文无差别写进长期记忆。此时更合适的做法是？

标准答案：D

解析：滚动摘要是“全保留”和“全丢弃”之间的折中方案。它的价值在于压缩长度，同时尽量保留后续判断真正依赖的背景信息，适合处理超长对话中的短期连续性问题。

错项分析：A 会直接丢失关键上下文；B 会把短期问题粗暴推给长期记忆，制造大量噪声；C 让用户反复概括成本过高，也不稳定。

知识点：滚动摘要与短期记忆管理

考纲映射：记忆管理 -> 摘要 / 上下文压缩

### 10. 【基础·3.5】某团队每次做代码审查都临时拼接很长 Prompt，不同成员写法差异很大，结果同样的代码在不同人手里得到完全不同的审查深度。若目标是沉淀可复用、可共享、可演进的能力，最适合的做法是？

标准答案：A

解析：题干要解决的是流程可复用和团队一致性。把稳定的审查流程沉淀为 Skill，可以把触发条件、步骤、资源和预期输出固定下来，比每次重新写 Prompt 更利于团队共享和后续迭代。

错项分析：B 仍然依赖个人发挥；C 只能保留短期连续性；D 长期记忆可以保留事实，但不能自动形成规范化流程单元。

知识点：Skill 作为团队能力沉淀方式

考纲映射：Skill 设计与沉淀 -> Prompt 到 Skill

### 11. 【中档·3.6】某 Agent 系统上线后整体成功率下降。团队怀疑问题可能出在“检索质量下降”“工具调用参数退化”或“生成阶段引用错证据”中的某一处，但仅看最终回答很难区分。此时优先应该做什么？

标准答案：C

解析：如果无法确定退化点在哪一层，就应该先把链路拆开做白盒评测。这样才能分别观察检索、工具调用和生成阶段的表现，避免把不同模块的问题混在一起。

错项分析：A 只扩大端到端样本，仍然看不到内部原因；B 一次替换多个模块会进一步放大归因难度；D 纯经验判断缺少可验证性。

知识点：白盒评测与模块化归因

考纲映射：评测驱动开发 -> 白盒评测

### 12. 【拔高·综合】某团队要做一个可持续迭代的代码审查 Agent，要求它能读文件、调用搜索、遵循团队审查规范、跨会话记住项目约束，并且每次升级后都能验证质量是否真的变好。以下哪组能力组合最完整？

标准答案：B

解析：题干是一个典型的工程化闭环题。读文件和搜索对应工具调用，团队规范沉淀对应 Skill，跨会话项目约束对应持久记忆，升级验证对应评测闭环。四者合起来才构成可持续迭代的系统。

错项分析：A 只有更强模型和主观复查，缺少规范沉淀与持久记忆；C 还是临时拼接；D 几乎没有真正的工程闭环。

知识点：工具、Skill、记忆与评测的组合设计

考纲映射：综合能力 -> Agent 工程闭环

## 二、多选题答案解析

### 13. 【基础·3.1】某 Agent 接入“创建日历事件”工具后，偶尔会输出字段名拼错、时间格式不对或漏掉参会人列表。为了提升工具调用可靠性，以下哪些做法更合理？

标准答案：ABC

解析：高质量工具调用可靠性通常包括三层：先用 schema 约束生成空间，再在程序侧校验，最后把校验失败原因反馈给模型重试。这三层组合起来，才能形成稳定的自动化链路。

错项分析：D 看似给模型更大自由，但题干的问题恰恰来自“自由输出导致程序不可消费”，因此它与目标相反。

知识点：结构化输出、程序校验与重试闭环

考纲映射：Agent 运行机制与工具调用 -> 校验 / 重试 / 结构化输出

### 14. 【中档·3.2】关于工作流、规划和反馈机制，以下哪些说法更合理？

标准答案：ACD

解析：A 讲的是固定流程的适用边界，C 讲的是运行时错误更依赖真实反馈，D 讲的是高风险任务更需要显式拆解和中间状态。三者都强调“根据任务属性选机制”，而不是一把梭。

错项分析：B 错在“只要复杂就一定更优”这种绝对化判断。Plan & Execute 很有用，但并不天然优于固定工作流，尤其在流程稳定、合规要求高的任务里并非首选。

知识点：固定工作流、显式计划与外部反馈的边界

考纲映射：工作流、规划与反思 -> 任务适配 / 反馈机制

### 15. 【基础·3.3】关于多 Agent 协作模式，以下哪些判断是正确的？

标准答案：ABD

解析：A 描述的是 Leader-Worker，B 描述的是 Blackboard，D 描述的是 handoff。三者都是常见协作方式，只是组织逻辑不同：中心调度、共享空间、阶段性交接。

错项分析：C 错在把 handoff 误说成“始终由同一个总控角色持续调度”，那其实更接近 Leader-Worker，而不是 handoff。

知识点：Leader-Worker、Blackboard、handoff 的区别

考纲映射：多 Agent 协作模式 -> 组织方式与边界

### 16. 【中档·3.4】关于 Agent 的记忆管理，以下哪些做法是合理的？

标准答案：ABC

解析：A 解决短期窗口占位问题，B 解决长对话连续性问题，C 解决跨会话复用问题。它们不是互斥关系，而是面向不同时间尺度的组合策略。

错项分析：D 错在把“全部永久保存”当成长期记忆治理。长期记忆的难点从来不是“能不能存”，而是“存什么、怎么召回、如何防噪声”。

知识点：短期记忆与长期记忆的组合治理

考纲映射：记忆管理 -> 截断 / 摘要 / 长期记忆

### 17. 【中档·3.5】关于 Skill 的设计与演进，以下哪些说法更合理？

标准答案：BCD

解析：高质量 Skill 不是简单把高频任务封起来，而是要先明确何时触发、依赖什么资源、如何输入输出，并支持版本化演进。因此 B、C、D 都成立。

错项分析：A 错在过于草率。高频并不等于已经适合沉淀为 Skill；如果触发条件和边界都不清楚，过早固化反而会放大误触发和维护成本。

知识点：Skill 的触发边界、依赖声明与版本治理

考纲映射：Skill 设计与沉淀 -> 触发条件 / 版本化

### 18. 【中档·3.6】关于评测驱动 Agent 开发，以下哪些说法更合理？

标准答案：ABD

解析：评测驱动强调不能只凭感觉判断，端到端评测关注最终用户价值，而长期回归集则防止同类退化反复出现。A、B、D 共同构成“判断结果是否变好”这件事的基础。

错项分析：C 错在把评测集当成一次性资产。系统持续演进后，回归集也需要持续维护和扩充，而不是一劳永逸冻结。

知识点：端到端评测与长期回归集

考纲映射：评测驱动开发 -> 持续评测 / 回归集建设

### 19. 【基础·3.7】关于 Qwen Code 的工作方式，以下哪些说法更合理？

标准答案：ABC

解析：`Plan Mode` 适合先分析方案，`/compress` 适合压缩长上下文，`/clear` 适合主动切换任务边界。这三项都对应真实的上下文管理和工作模式选择。

错项分析：D 错在把 EPCC 的 `Explore` 理解成“先改再理解”。`Explore` 的重点恰恰是先摸清现状和边界，再决定后续动作。

知识点：Qwen Code 的模式与上下文管理

考纲映射：Qwen Code 实践 -> Plan Mode / 上下文管理 / EPCC

### 20. 【拔高·综合】某团队希望打造一个能持续优化的 Agent 开发流程，要求既能接入外部能力，又能沉淀团队规范、记住长期约束，并持续验证系统是否真正变好。以下哪些设计更合理？

标准答案：ACD

解析：A 负责接入外部能力，C 负责承载长期项目约束，D 负责持续验证和定位升级效果。这三者共同构成工程化闭环。题干并不是反对系统提示词，而是反对“只靠一次性 Prompt 堆规范、后续不再演进”的做法。

错项分析：B 错在把团队规范长期停留在一次性系统提示词里，不沉淀成可复用、可版本化的 Skill，会让规范维护越来越脆弱。

知识点：外部能力、长期记忆与评测闭环的组合

考纲映射：综合能力 -> 工具 / Skill / 记忆 / 评测

## 三、补充单选题答案解析

### 21. 【中档·3.1】某 Agent 接入“创建工单”工具后，偶尔因为网络抖动而超时重试，结果同一个请求被系统创建出两张重复工单。排查发现参数大多是正确的。若只能优先补一项机制，最应该先做什么？

标准答案：D

解析：这道题考的是写操作的幂等治理，而不是参数格式问题。既然参数本身大多正确，真正的风险来自“同一意图被重复执行”，因此优先应该在工具边界增加幂等键或去重机制。

错项分析：A 和 C 都停留在语言层，不能阻止重复写入；B 彻底关闭重试虽然能减少重复，但会显著降低系统对瞬时失败的恢复能力，不是更稳妥的第一步。

知识点：写操作工具的幂等与去重

考纲映射：Agent 运行机制与工具调用 -> 写操作治理

### 22. 【中档·3.2】以下哪类任务最不适合优先交给 Agent 做完全自主规划，而更适合预设固定工作流？

标准答案：B

解析：报销审批流步骤固定、错误代价高、合规要求明确，因此更需要显式流程和校验，而不是把路径选择完全交给 Agent 自主探索。

错项分析：A、C、D 都更偏探索性或开放性任务，反而更适合让 Agent 生成候选路线或方案。

知识点：自主规划与固定工作流的适用边界

考纲映射：工作流、规划与反思 -> 任务分类与可控性

### 23. 【中档·3.3】某系统由三个专业 Agent 顺序协作：需求澄清 Agent 先完成上下文整理，再把结论交给实现 Agent；实现 Agent 处理完后，再把结果移交给测试 Agent。整个流程没有一个中心角色持续拆解和调度全部参与者。这个模式更接近？

标准答案：A

解析：题干强调的是“阶段性交权”，而不是“中心调度者持续分配任务”。这就是 handoff 的核心特征：一个角色完成自己的阶段目标后，把上下文和控制权移交给下一个角色。

错项分析：B 需要明显的中心调度者；C 强调共享中间状态；D 只是信息存储或召回方式，不是协作模式。

知识点：handoff 与 Leader-Worker 的边界

考纲映射：多 Agent 协作模式 -> handoff

### 24. 【中档·3.4】某团队把所有对话原文都写入长期记忆，结果记忆库越来越大，但召回出的内容经常无关、过时，甚至和当前项目阶段冲突。更合理的改进是？

标准答案：C

解析：长期记忆的关键不是“存满”，而是“写准”。更合理的治理方式是只写入稳定、高价值、跨会话可复用的信息，并为写入过程设置筛选规则和时效判断。

错项分析：A 只解决容量，不解决噪声；B 走向另一个极端，放弃长期记忆价值；D 清空全部记忆虽然能消噪，但也丢掉真正有复用价值的信息。

知识点：长期记忆写入策略与噪声控制

考纲映射：记忆管理 -> 长期记忆治理

## 四、补充多选题答案解析

### 25. 【中档·3.5】关于 Skill 的触发边界与版本演进，以下哪些说法更合理？

标准答案：ACD

解析：A 说明 Skill 更适合高频、稳定、可复用流程；C 说明版本控制能支持回滚、对比和团队协作；D 说明高质量 Skill 需要清晰说明输入输出、触发条件和资源依赖。这三项共同决定了 Skill 是否可维护。

错项分析：B 错在把模糊触发当成覆盖面优势。触发条件越模糊，越容易误触发、漏触发，最终削弱 Skill 的稳定性。

知识点：Skill 的触发边界与版本演进

考纲映射：Skill 设计与沉淀 -> 触发条件 / 版本治理

### 26. 【拔高·3.6】关于评测闭环如何指导下一轮 Agent 优化，以下哪些做法更合理？

标准答案：ABC

解析：评测闭环的价值在于指导下一轮行动，而不是只产出一张成绩单。A 负责定位问题落点，B 负责把结果真正反馈到策略设计，C 负责把反复失败样例沉淀为回归资产。

错项分析：D 错在把用户投诉当成唯一信号。离线评测往往能更早发现系统退化，不能因为“暂时没人报错”就忽略。

知识点：评测闭环、白盒定位与回归集建设

考纲映射：评测驱动开发 -> 闭环优化 / 回归测试

## 考点覆盖与复习建议

- 先复习工具调用中的结构化约束、程序校验和写操作幂等治理，建立“能调用”和“能稳定调用”的区别。
- 再复习固定工作流、显式计划、外部反馈与多 Agent 协作模式，重点区分它们的适用边界。
- 然后复习短期上下文管理、滚动摘要、长期记忆写入规则，理解不同记忆机制面向的时间尺度。
- 最后把 Skill、Qwen Code 方法和评测闭环串起来，形成完整的 Agent 工程思维。
