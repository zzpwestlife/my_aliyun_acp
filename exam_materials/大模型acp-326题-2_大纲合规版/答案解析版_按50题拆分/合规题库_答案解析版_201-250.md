#### 第 201 题 [多选题]

以下哪些场景适合使用流式对话 LLM_MODEL.stream_chat (messages)？

A. 实时聊天机器人，需要立即响应用户输入。

B. 需要对大模型的输出进行实时监控和干预的场景。

C. 用户需要快速获得初步结果，后续可以继续补充信息完善结果。

D. 只需要一次性返回完整结果的简单任务，例如翻译单个单词。

E. 需要处理长文本输入的场景，可以逐步返回结果，避免超时。

**答案：`ABCE`**

**考点：** 大模型应用开发 / API 调用参数、消息角色与对话历史

**详细解析：**

**相关知识点：** API 调用题要分清 model、messages、system/user/assistant、temperature、top_p、stream、seed、max_tokens 等参数各自作用。 API = 参数、消息角色、流式/批量、对话历史和适用边界。


**原解析补充：**

D 简单任务无实时交互需求，流式反而增加复杂度

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：实时聊天机器人，需要立即响应用户输入。
  - 原因：流式输出（stream_chat）能边生成边推送 token，用户无需等待完整响应，契合实时聊天的低延迟需求。
- B 正确：需要对大模型的输出进行实时监控和干预的场景。
  - 原因：流式接口逐 token 返回，调用方可在每个 chunk 上实时检查并中断或修改输出，满足监控与干预需求。
- C 正确：用户需要快速获得初步结果，后续可以继续补充信息完善结果。
  - 原因：流式模式让用户看到部分结果后即可开始阅读和反馈，不必等全文生成完毕，符合迭代补充的交互场景。
- D 不选：只需要一次性返回完整结果的简单任务，例如翻译单个单词。
  - 原因：翻译单个单词响应极短，流式输出带来额外复杂度却无收益，一次性批量调用更高效。
- E 正确：需要处理长文本输入的场景，可以逐步返回结果，避免超时。
  - 原因：长文本生成耗时较长，流式模式持续推送 token 可绕开接口超时限制，用户也能逐步阅读结果。

**易错点分析：**

易错点：temperature/top_p 控制随机性，max_tokens 控制长度，stream 控制返回方式，seed 控制一致性，system 不是用户提问。

#### 第 202 题 [多选题]

以下哪些措施可以提高大模型在私域场景下的安全性？

A. 对相关人员进行安全培训

B. 建立完善的安全 incident 响应机制

C. 使用开源的数据管理软件

D. 将所有数据存储在云端

E. 定期更新大模型和相关软件

F. 对大模型进行持续的安全评估和测试

**答案：`ABEF`**

**考点：** 生产环境应用实践 / 内容安全、合规检查与应用安全

**详细解析：**

**相关知识点：** 生产安全题优先考虑 API Key 安全、输入过滤、输出审核、权限控制、隐私授权、备案、日志追踪和风险兜底。 Production = 安全、合规、部署、性能、成本、监控、降级。


**原解析补充：**

核心措施：人员培训（A）+ 应急机制（B）+ 持续更新（E）+ 安全测试（F）需避免：盲目依赖开源（C）或云端（D），需结合业务需求定制方案

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：对相关人员进行安全培训
  - 原因：正确，因为它符合生产环境的安全合规要求，可以降低泄漏、违规或上线风险。
- B 正确：建立完善的安全 incident 响应机制
  - 原因：正确，因为它符合生产环境的安全合规要求，可以降低泄漏、违规或上线风险。
- C 不选：使用开源的数据管理软件
  - 原因：开源软件是否安全取决于具体选型与维护，不构成通用安全措施；且私域安全重点在访问控制与合规，而非软件是否开源。
- D 不选：将所有数据存储在云端
  - 原因："将所有数据存储在云端"过于绝对，私域场景可能有数据主权要求，盲目上云反而带来数据泄露风险。
- E 正确：定期更新大模型和相关软件
  - 原因：及时打安全补丁、升级依赖，能修复已知漏洞，减少被攻击面，是持续维护私域系统安全的基本措施。
- F 正确：对大模型进行持续的安全评估和测试
  - 原因：定期红队测试和安全评估能发现对抗性输入、越权调用等新风险，持续保障模型在私域环境的行为边界。

**易错点分析：**

易错点：“先上线后备案”“直接返回风险内容”“把 Key 写进代码”通常都是错误方向。

#### 第 203 题 [多选题]

Assistant API 的 Assistant 类的各项功能中涵盖了以下哪些操作？

A. 删除

B. 更新

C. 剪切

D. 创建

E. 复制

F. 列举

**答案：`ABDF`**

**考点：** 多Agent及多模态应用 / Assistant API、智能体机制与工具调用

**详细解析：**

**相关知识点：** Assistant/Agent 题重点看 Thread、Message、Run、工具函数、函数参数、submit outputs、Planner 路由和任务拆解。 Agent = 规划和工具调用；多模态 = 跨文本、图像、语音、视频。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：删除
  - 原因：Assistant API 提供 `assistants.delete()` 方法，支持删除已有 Assistant 对象，属于标准 CRUD 操作。
- B 正确：更新
  - 原因：`assistants.update()` 允许修改 Assistant 的名称、指令、工具配置等属性，是 API 的核心管理操作之一。
- C 不选：剪切
  - 原因：剪切（Cut）是文本编辑器概念，Assistant API 不提供此操作；对资源的迁移或移动需通过创建+删除组合完成。
- D 正确：创建
  - 原因：`assistants.create()` 是最基础的 API 操作，用于初始化一个携带指令和工具配置的 Assistant 对象。
- E 不选：复制
  - 原因：Assistant API 没有原生的复制（Clone）接口；若需复制，须手动读取配置再调用 create，并非 API 内置操作。
- F 正确：列举
  - 原因：`assistants.list()` 可分页列出账户下所有 Assistant，方便批量管理和查询，是标准管理操作。

**易错点分析：**

易错点：工具调用错误通常先查工具描述、参数 schema 和路由逻辑，不要先盲目换大模型。

#### 第 204 题 [多选题]

在监控大语言模型训练性能时，哪些指标是常用的？

A. 训练时间

B. 生成速度

C. batch size

D. 精准率

E. 损失函数值

F. 准确率

**答案：`ABE`**

**考点：** 大模型微调 / 微调概念、数据构建、训练参数与评测

**详细解析：**

**相关知识点：** 微调是用任务数据继续训练预训练模型，核心关注数据质量、训练循环、损失函数、梯度、学习率、batch size、epoch、验证集和过拟合/欠拟合。 Fine-tuning = 用任务数据更新模型参数，适合稳定任务适配。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：训练时间
  - 原因：训练时间直接衡量硬件利用效率和整体训练进度，是衡量大模型训练性能的核心运行指标。
- B 正确：生成速度
  - 原因：生成速度（tokens/s）反映模型推理吞吐量，是监控训练期间模型生成能力的常用性能指标。
- C 不选：batch size
  - 原因：batch size 是超参数设置而非监控指标，它影响收敛速度但本身不作为训练过程中"监控"的性能量。
- D 不选：精准率
  - 原因：精准率（Precision）是分类任务的评估指标，大模型生成式训练通常用 loss 而非精准率监控训练过程。
- E 正确：损失函数值
  - 原因：loss 是衡量每个批次模型预测与目标差距的核心量，training loss 下降趋势是训练是否正常进行的直接依据。
- F 不选：准确率
  - 原因：准确率（Accuracy）常用于分类任务评估阶段，生成式大模型训练过程主要监控 loss，而非 accuracy。

**易错点分析：**

易错点：微调不是知识库更新的低成本替代；频繁变化的事实知识通常优先用 RAG。训练题要分清 loss、backward、step 的职责。

#### 第 205 题 [单选题]

你想使用 Qwen-Max 从一段冗长的课程文本中提炼出关键信息，用于创建 PPT 脚本。以下哪个提示词最合适？

A. 用更少的字概括这段文字

B. 缩短这段文字

C. 从以下课程文本中提取关键信息，生成一个简洁的 PPT 脚本大纲，突出重点内容，并控制在5 个要点以内。

D. 把这段文字变成 PPT

**答案：`C`**

**考点：** 大模型提示词工程 / 提示词框架、系统角色与任务设计

**详细解析：**

**相关知识点：** 提示词工程的核心是把角色、任务、上下文、约束、输出格式、样例写清楚；好的提示词通常具体、可执行、可解析。 Prompt = 把角色、任务、上下文、约束、输出格式和样例讲清楚。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 不选：用更少的字概括这段文字
  - 原因：只说"更少的字"没有明确任务目标（PPT脚本），也没有约束输出格式和要点数量，模型无法准确定向输出。
- B 不选：缩短这段文字
  - 原因："缩短"仅描述了压缩操作，未指定输出形式（PPT大纲）和结构要求，与提炼 PPT 脚本的目标不匹配。
- C 正确：从以下课程文本中提取关键信息，生成一个简洁的 PPT 脚本大纲，突出重点内容，并控制在5 个要点以内。
  - 原因：该提示词明确指定了任务（提取关键信息）、输出形式（PPT脚本大纲）和约束（5个要点以内），符合具体、可执行的高质量提示词标准。
- D 不选：把这段文字变成 PPT
  - 原因："变成 PPT"表述模糊，未说明输出格式、要点结构或内容侧重，模型难以生成符合预期的脚本大纲。

**易错点分析：**

易错点：只写“请做好/请准确”不算好提示词。考试更偏向选择具体、结构化、带格式和样例的表达。

#### 第 206 题 [单选题]

在设定大语言模型的训练参数时，哪个参数可以影响模型每次更新时所考虑的数据量，并间接影响到模型收敛速度和内存使用？

A. 学习率

B. 正则化

C. 批大小

D. 动态学习率策略

**答案：`C`**

**考点：** 大模型微调 / 微调概念、数据构建、训练参数与评测

**详细解析：**

**相关知识点：** 微调是用任务数据继续训练预训练模型，核心关注数据质量、训练循环、损失函数、梯度、学习率、batch size、epoch、验证集和过拟合/欠拟合。 Fine-tuning = 用任务数据更新模型参数，适合稳定任务适配。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 不选：学习率
  - 原因：学习率控制每次梯度更新的步长，影响收敛速度，但不决定每次更新所处理的数据量，不符合题目描述。

  - 原因：正则化（如 L1/L2）用于防止过拟合，限制权重大小，与"每次更新考虑的数据量"和内存占用无直接关系。

  - 原因：动态学习率策略（如 warmup、余弦退火）调整的是学习率随训练步骤的变化，与每次处理的数据量无关。

#### 第 207 题 [单选题]

StorageContext.from_defaults (persist_dir=PERSIST_DIR) 的作用是什么？

A. 创建一个新的 StorageContext 对象，并使用默认配置，但指定持久化目录。

B. 将 StorageContext 对象保存到 PERSIST_DIR。

C. 从 PERSIST_DIR 加载 StorageContext 对象。

D. 创建一个默认的 StorageContext 对象，并忽略 PERSIST_DIR。

**答案：`A`**

**考点：** 大模型检索增强 / RAG 核心流程、检索优化、文档解析/切片

**详细解析：**

**相关知识点：** RAG 的关键链路是：文档解析 -> 切片/补上下文 -> embedding 向量化 -> 向量库召回 -> rerank/过滤 -> 拼接上下文生成。答题时先定位问题发生在哪一层。 RAG = 外部知识 + 检索相关片段 + 拼接上下文 + 生成答案。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：创建一个新的 StorageContext 对象，并使用默认配置，但指定持久化目录。
  - 原因：`from_defaults(persist_dir=...)` 是工厂方法，使用默认的向量库、文档库等配置创建新对象，同时将 persist_dir 设为该目录，后续调用 `storage_context.persist()` 时数据将写入此处。
- B 不选：将 StorageContext 对象保存到 PERSIST_DIR。
  - 原因：该方法仅是创建对象时指定持久化目录，实际保存操作需要显式调用 `.persist()` 方法，两者不同。
- C 不选：从 PERSIST_DIR 加载 StorageContext 对象。
  - 原因：从已有目录加载应使用 `StorageContext.from_defaults(persist_dir=PERSIST_DIR)` 配合 `load_index_from_storage()`，但该方法本身是创建而非加载，C 描述混淆了两个步骤。
- D 不选：创建一个默认的 StorageContext 对象，并忽略 PERSIST_DIR。
  - 原因：传入 `persist_dir` 参数后不会被忽略，该参数会被记录到 StorageContext 对象中作为后续持久化的目标路径。

**易错点分析：**

易错点：不要把所有问题都归因于“大模型不够强”。如果问题出在文档解析、切片、召回或重排，调 temperature 或换生成模型通常不是首选。
