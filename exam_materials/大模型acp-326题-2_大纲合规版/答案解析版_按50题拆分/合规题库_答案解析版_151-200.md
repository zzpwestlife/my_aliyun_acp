#### 第 151 题 [单选题]

当我们发现使用的模型在大数计算时表现不佳，哪个措施提升最大？

A. 改进提示词

B. 微调模型，让它学会在大数计算时使用计算器插件

C. 提供大数计算样例

D. 使用微调提高模型的逻辑推理能力

**答案：`B`**

**考点：** 大模型微调 / 微调概念、数据构建、训练参数与评测

**详细解析：**

**相关知识点：** 微调是用任务数据继续训练预训练模型，核心关注数据质量、训练循环、损失函数、梯度、学习率、batch size、epoch、验证集和过拟合/欠拟合。 Fine-tuning = 用任务数据更新模型参数，适合稳定任务适配。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 不选：改进提示词
  - 原因：大数计算失败是模型能力边界问题，不是提示词表达问题；改进提示词无法让模型突破底层计算局限。
- B 正确：微调模型，让它学会在大数计算时使用计算器插件
  - 原因：通过微调让模型学会调用计算器插件，是从根本上弥补大数计算短板的方式，直接解决能力缺口。
- C 不选：提供大数计算样例
  - 原因：few-shot 样例只是提示词层面的辅助，仅提供样例而不让模型工具化，无法保证复杂大数计算的准确率。
- D 不选：使用微调提高模型的逻辑推理能力
  - 原因：大数计算失败的根因是精确数值计算，而非逻辑推理；微调逻辑推理不能直接修复数值计算精度问题。

**易错点分析：**

易错点：微调不是知识库更新的低成本替代；频繁变化的事实知识通常优先用 RAG。训练题要分清 loss、backward、step 的职责。

**原题号：** 159

#### 第 152 题 [单选题]

某公司更新了组织架构，但知识库还是旧文档，查询结果经常过时。为避免回答陈旧信息，最合适的方式是？

A. 定期同步最新文档并重建索引

B. 调整 Node Parser 切片长度

C. 对旧知识不做任何处理

D. 增大模型温度让其自由生成

**答案：`A`**

**考点：** 大模型检索增强 / RAG 核心流程、检索优化、文档解析/切片

**详细解析：**

**相关知识点：** RAG 的关键链路是：文档解析 -> 切片/补上下文 -> embedding 向量化 -> 向量库召回 -> rerank/过滤 -> 拼接上下文生成。答题时先定位问题发生在哪一层。 RAG = 外部知识 + 检索相关片段 + 拼接上下文 + 生成答案。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：定期同步最新文档并重建索引
  - 原因：知识库陈旧是因为索引内容未及时更新；定期同步最新文档并重建索引直接清除过时信息，从数据源头解决回答陈旧的问题。
- B 不选：调整 Node Parser 切片长度
  - 原因：切片长度影响的是检索粒度，而不是文档内容是否最新；组织架构更新后，旧切片换个粒度仍是过时内容。
- C 不选：对旧知识不做任何处理
  - 原因：不处理旧文档意味着知识库继续返回过时答案，完全没有解决查询结果陈旧的问题。
- D 不选：增大模型温度让其自由生成
  - 原因：temperature 影响的是生成多样性，不能给模型注入最新的组织架构信息；提高温度只会让回答更随意，而非更准确。

**易错点分析：**

易错点：不要把所有问题都归因于“大模型不够强”。如果问题出在文档解析、切片、召回或重排，调 temperature 或换生成模型通常不是首选。

#### 第 153 题 [单选题]

某企业计划通过阿里云百炼平台构建一个大模型应用，用于处理对实时性要求不高的任务（如离线招生生成、批量内容摘要提取等）。为了在满足业务需求的同时降低运行成本，以下哪种措施可以最有效地优化成本？

A. 引入流式输出，降低模型推理成本

B. 调用最新且参数大的模型（如 Qwen-max）通过更好的技术降低计算成本

C. 使用批量推理服务，离线方式处理数据

D. 调用支持上下文缓存（Context Cache）的模型减少重复请求的计算开销

**答案：`C`**

**考点：** 生产环境应用实践 / 部署发布、性能与生产实践

**详细解析：**

**相关知识点：** 生产实践题重点看延迟、成本、稳定性、降级、缓存、批处理、流式输出、资源限制和工程可维护性。 Production = 安全、合规、部署、性能、成本、监控、降级。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 不选：引入流式输出，降低模型推理成本
  - 原因：流式输出改变的是响应返回方式（边生成边推送），不会减少模型实际计算量，也不降低 token 消耗成本。
- B 不选：调用最新且参数大的模型（如 Qwen-max）通过更好的技术降低计算成本
  - 原因：更大参数的模型单次推理成本更高而非更低；用高价模型处理对实时性要求不高的批量任务，与降本目标相悖。
- C 正确：使用批量推理服务，离线方式处理数据
  - 原因：批量推理服务专为非实时离线场景设计，可在资源空闲时段统一处理，单位成本显著低于实时在线调用。
- D 不选：调用支持上下文缓存（Context Cache）的模型减少重复请求的计算开销
  - 原因：上下文缓存适合同一上下文多次复用的场景（如长对话），对一次性离线摘要生成等任务帮助有限，并非最优降本手段。

**易错点分析：**

易错点：流式输出改善用户等待体验，但不等于降低模型计算成本；缓存/批处理/降级才常用于成本和稳定性优化。

**原题号：** 161

#### 第 154 题 [单选题]

在大模型调用外部 API 工具时，工具函数的描述（description）起到的关键作用是什么？

A. 帮助大模型理解何时以及如何调用该函数

B. 减少 API 调用过程中的内存占用

C. 加密 API 请求中的敏感数据

D. 提高 API 调用的速度

**答案：`A`**

**考点：** 生产环境应用实践 / 内容安全、合规检查与应用安全

**详细解析：**

**相关知识点：** 生产安全题优先考虑 API Key 安全、输入过滤、输出审核、权限控制、隐私授权、备案、日志追踪和风险兜底。 Production = 安全、合规、部署、性能、成本、监控、降级。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：帮助大模型理解何时以及如何调用该函数
  - 原因：工具函数的 description 字段是模型决策"是否调用"和"如何传参"的依据；清晰的描述直接决定工具调用的准确性。
- B 不选：减少 API 调用过程中的内存占用
  - 原因：description 是文本字段，提供语义说明，与运行时内存占用毫无关联；减少内存是底层系统层面的优化。
- C 不选：加密 API 请求中的敏感数据
  - 原因：description 是工具语义描述，不涉及数据加密；加密是通过 TLS/HTTPS 或密钥管理实现的，与函数描述无关。
- D 不选：提高 API 调用的速度
  - 原因：description 不影响网络延迟或推理速度；提高速度依赖模型规模、网络带宽和推理优化，与函数描述内容无关。

**易错点分析：**

易错点：“先上线后备案”“直接返回风险内容”“把 Key 写进代码”通常都是错误方向。

#### 第 155 题 [单选题]

梯度下降算法中，如果学习率设置过大可能导致？

A. 在最优解附近震荡

B. 需要更多迭代次数才能收敛

C. 收敛速度过慢

D. 参数更新幅度不足

**答案：`A`**

**考点：** 大模型微调 / 微调概念、数据构建、训练参数与评测

**详细解析：**

**相关知识点：** 微调是用任务数据继续训练预训练模型，核心关注数据质量、训练循环、损失函数、梯度、学习率、batch size、epoch、验证集和过拟合/欠拟合。 Fine-tuning = 用任务数据更新模型参数，适合稳定任务适配。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：在最优解附近震荡
  - 原因：学习率过大时每步更新幅度过大，会跨越最优点来回反弹，导致损失函数在最优解附近持续震荡而无法稳定收敛。
- B 不选：需要更多迭代次数才能收敛
  - 原因：迭代次数更多是学习率过小时的后果，学习率过大反而是每步迈过了最优解，问题不是"太慢"而是"不稳"。
- C 不选：收敛速度过慢
  - 原因：收敛过慢同样是学习率过小的症状；学习率过大会使参数跳跃过大，速度不慢但会错过并反复穿越最优解。
- D 不选：参数更新幅度不足
  - 原因：参数更新幅度不足是学习率过小的表现，与"学习率过大"方向相反；学习率过大的问题恰好是更新幅度过大。

**易错点分析：**

易错点：微调不是知识库更新的低成本替代；频繁变化的事实知识通常优先用 RAG。训练题要分清 loss、backward、step 的职责。

#### 第 156 题 [单选题]

你需要借助大模型为小学生写一篇关于 “保护环境” 的科普短文，但输出的内容过于学术化（频繁使用 “碳达峰”“生物降解” 等术语）。以下哪个提示词优化方向能最直接解决该问题？

A. 定义角色与受众：以 “小学科学老师” 身份，用比喻和故事向 “6 - 12 岁学生” 讲解

B. 提供样例：附上一篇《如何节约用水》的短文（语言平实，包含互动问答）

C. 补充任务目标：要求文章包含 “垃圾分类对社区的影响” 案例

D. 扩展上下文：添加背景 “当前城市垃圾填埋占比超过 60%”

**答案：`A`**

**考点：** 大模型提示词工程 / 提示词框架、系统角色与任务设计

**详细解析：**

**相关知识点：** 提示词工程的核心是把角色、任务、上下文、约束、输出格式、样例写清楚；好的提示词通常具体、可执行、可解析。 Prompt = 把角色、任务、上下文、约束、输出格式和样例讲清楚。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 正确：定义角色与受众：以 “小学科学老师” 身份，用比喻和故事向 “6 - 12 岁学生” 讲解
  - 原因：内容过于学术化的根因是模型不知道目标受众；明确”小学科学老师”角色和”6-12 岁学生”受众，直接约束输出语言风格为通俗易懂。
- B 不选：提供样例：附上一篇《如何节约用水》的短文（语言平实，包含互动问答）
  - 原因：样例法（few-shot）也能间接引导语言风格，但解决”过于学术化”的最直接手段是明确角色与受众定义，而非示范另一主题的文章。
- C 不选：补充任务目标：要求文章包含 “垃圾分类对社区的影响” 案例
  - 原因：补充案例内容方向只是扩充了文章主题，并未限制语言难度；模型仍可能用学术术语来写”垃圾分类”案例。
- D 不选：扩展上下文：添加背景 “当前城市垃圾填埋占比超过 60%”
  - 原因：添加统计数据背景属于信息补充，不能指导模型以儿童语言表达；反而可能引导模型输出更多数据化、学术化内容。

**易错点分析：**

易错点：只写“请做好/请准确”不算好提示词。考试更偏向选择具体、结构化、带格式和样例的表达。

#### 第 157 题 [单选题]

以下哪种方式有助于在大模型多轮对话场景中，保障生成内容质量的同时，降低成本，并提升响应速度？

A. 使用 Batch API

B. 并行调用大模型 API

C. 使用支持上下文缓存的大模型 API

D. 使用经过量化的模型

**答案：`C`**

**考点：** 大模型微调 / 微调概念、数据构建、训练参数与评测

**详细解析：**

**相关知识点：** 微调是用任务数据继续训练预训练模型，核心关注数据质量、训练循环、损失函数、梯度、学习率、batch size、epoch、验证集和过拟合/欠拟合。 Fine-tuning = 用任务数据更新模型参数，适合稳定任务适配。

**解题逻辑：** 先看题目问的是“最合适/不推荐/主要作用/哪些措施”，再定位到具体环节。单选题通常选最直接解决根因的一项；多选题则保留所有能直接改善该环节的措施，排除只改善无关指标、过度泛化或跳过必要流程的选项。

**逐选项原因：**
- A 不选：使用 Batch API
  - 原因：Batch API 是异步批量处理，适合降低非实时任务成本，但无法保障多轮对话的内容质量，也不能提升响应速度。
- B 不选：并行调用大模型 API
  - 原因：并行调用可缩短部分场景的等待时间，但会增加成本，且多轮对话的上下文是顺序依赖的，无法简单并行。
- C 正确：使用支持上下文缓存的大模型 API
  - 原因：多轮对话中系统提示和历史消息会被反复输入；上下文缓存对重复前缀 token 不重复计费、不重复计算，同时降低成本并提升响应速度，且不影响生成质量。
- D 不选：使用经过量化的模型
  - 原因：量化模型可降低推理资源占用，但会损失一定模型精度，无法保障"生成内容质量"这一前提，且需自行部署。

**易错点分析：**

易错点：微调不是知识库更新的低成本替代；频繁变化的事实知识通常优先用 RAG。训练题要分清 loss、backward、step 的职责。

#### 第 158 题 [单选题]

你使用阿里云百炼平台的通义千问 API 开发了聊天机器人，用户反馈：提问某地天气时模型总是无法回答。如何快速解决这个问题？

A. 提前设置话术，在面对类似问题时可以直接将话术返回给用户

B. 通过复杂的工程手段，对用户的输入提取关键词，集成第三方搜索引擎，并将搜索结果返回给大模型

C. 设置 enable_search（联网搜索）参数，为通义千问 API 引入联网能力

D. 在聊天界面显眼位置标注限制，不建议用户输入类似问题

**答案：`C`**

**考点：** 通义千问 API / 联网搜索 enable_search、实时信息增强

**详细解析：**

**相关知识点：** 通用大模型的参数知识有时效性限制，天气、新闻、价格、赛事等实时问题通常需要外部搜索或工具增强。百炼/通义千问 API 中如果提供 `enable_search` 这类联网搜索能力，最直接的做法是打开该能力，让模型在生成前获得实时信息。

**解题逻辑：** 题干关键词是“某地天气”和“快速解决”。天气属于实时信息，不是靠提示词话术、离线训练或复杂自建工程优先解决。考试里问“快速”时，通常选择平台已提供的开关或参数，而不是重新搭建搜索系统。

**逐选项原因：**
- A 不选：提前设置话术，在面对类似问题时可以直接将话术返回给用户
  - 原因：固定话术只能回避问题，不能真正回答实时天气；用户体验和问题解决度都不足。
- B 不选：通过复杂的工程手段，对用户的输入提取关键词，集成第三方搜索引擎，并将搜索结果返回给大模型
