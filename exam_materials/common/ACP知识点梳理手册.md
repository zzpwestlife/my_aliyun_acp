# 阿里云大模型ACP考试知识点梳理手册

> 依据《阿里云大模型高级工程师 ACP 认证考试大纲 V3》与《大模型ACP认证教程》(C1–C5) 编写。
> 配套文件：《ACP考点清单.md》《ACP模拟考试卷_含答案解析.md》。

## 如何使用本手册

- **权重**列：高（核心高频，必考）/ 中（常规，常考）/ 低（了解性，偶考）。综合模块占比与考纲强调度判定。
- **形式**列：识记（记概念/参数/名词）/ 理解（辨析机制与适用边界）/ 应用（操作、方案设计、故障排查）。映射自考纲动词：「了解」→识记，「理解」→理解，「掌握/动手/部署」→应用。
- **易错点**列：命题干扰项的来源，即考生最常见的认知误区与易混概念。

| 项 | 值 |
|---|---|
| 满分 / 及格 / 时长 | 100 分 / 80 分 / 120 分钟 |
| 题型 | 单选 50 题（各 1 分）+ 多选 25 题（各 2 分） |
| 模块占比 | 应用开发 17% · 提示词 15% · 检索增强 20% · 微调 16% · 多Agent多模态 16% · 生产实践 16% |

## 模块一 · 大模型应用开发（17%）

**教材来源**：C1_1 配置AI开发环境、C2_1 用大模型构建新人答疑机器人。
**模块概述**：通过 OpenAI 兼容方式调用百炼平台通义千问模型，掌握核心采样参数、消息结构、流式与批量生成，理解大模型「无状态、按 Token 自回归生成」的工作原理。

| 知识点 | 教材位置 | 权重 | 形式 | 常见易错点 |
|---|---|---|---|---|
| OpenAI 兼容调用百炼：`base_url`=`https://dashscope.aliyuncs.com/compatible-mode/v1` | C2_1 | 高 | 应用 | 误填 openai.com 或 bailian 域名；忘记 `/compatible-mode/v1` |
| API Key 安全存储（环境变量/配置文件加载，禁明文/禁入库） | C1_1,C2_1 | 中 | 应用 | 误以为写注释或私有仓库即安全 |
| `temperature`：控制随机性，越高越发散 | C2_1 | 高 | 理解 | 误以为它控制长度/质量；误以为可设负值 |
| `top_p`：核采样，从累计概率前 p 的 Token 中采样 | C2_1 | 高 | 理解 | 与 temperature 混淆；误以为 top_p=0.1 是「取概率最低」 |
| temperature 与 top_p 通常**择一调节**，不建议同时大幅改 | C2_1 | 中 | 理解 | 误以为两者必须同时调或互不相关 |
| `stream=True` 流式生成（打字机效果，降首字延迟） | C2_1 | 高 | 应用 | 误以为流式能提升答案质量或省 Token |
| 批量生成 vs 流式生成的适用场景 | C2_1 | 中 | 理解 | 把「批量」与「流式」对立，实为不同维度 |
| 消息角色 `system`/`user`/`assistant` 与对话历史拼接 | C2_1 | 高 | 理解 | 误以为 API 自动记忆历史 |
| 大模型 API 默认**无状态**：多轮需外部显式携带上下文 | C2_1,C3_4 | 高 | 理解 | 误以为模型自带长期记忆/缓存恢复 |
| 模型选型：qwen-max/plus/turbo 在能力—成本—延迟间权衡 | C2_1 | 中 | 应用 | 一律选 max，忽视成本/延迟 |
| 大模型工作原理：按 Token 自回归概率生成，可能幻觉 | C2_1 | 中 | 理解 | 误以为输出是检索数据库的确定答案 |

**高频考点提示**：base_url 正确值、temperature/top_p 辨析、stream 作用、无状态特性——几乎必考。

## 模块二 · 大模型提示词工程（15%）

**教材来源**：C2_3 优化提示词改善答疑机器人回答质量。
**模块概述**：用结构化提示词要素、分隔符、模板、系统角色提升回答质量；掌握 CoT/Few-shot 等技术与适用场景，了解 Meta Prompting、上下文工程等进阶方法。

| 知识点 | 教材位置 | 权重 | 形式 | 常见易错点 |
|---|---|---|---|---|
| 提示词要素：指令/上下文/输入/输出格式 | C2_3 | 高 | 理解 | 把「要素」与「分隔符」混为一谈 |
| 分隔符（`###`、`---`、`"""`、XML 标签）划分提示词区块 | C2_3 | 高 | 应用 | 误以为分隔符用于标识关键要素而非分区 |
| 提示词模板：变量占位，批量复用 | C2_3 | 中 | 应用 | 误以为模板=固定文本不可参数化 |
| 系统角色提示词（system）设定身份/风格/边界 | C2_3 | 高 | 理解 | 误以为 system 与 user 提示词效果等同 |
| Zero-shot vs Few-shot：是否给示例 | C2_3 | 高 | 理解 | 把 Few-shot 当成微调；混淆示例数量界定 |
| 思维链 CoT：引导逐步推理提升复杂任务正确率 | C2_3 | 高 | 应用 | 误以为 CoT 适合所有任务（简单任务徒增成本） |
| 思维树 ToT / 思维图 GoT：CoT 的扩展 | C2_3 | 中 | 识记 | 误归类为微调或多模态技术 |
| 大模型适用/不适用场景判断 | C2_3 | 中 | 应用 | 把需精确计算/强一致的任务交给模型直答 |
| 典型应用：意图分类、文档审阅、自动文档修订 | C2_3 | 中 | 应用 | 忽视输出格式约束导致难以解析 |
| Meta Prompting：用大模型迭代优化提示词本身 | C2_3 | 中 | 理解 | 误解为多图拼接/Token 压缩/Agent 通信协议 |
| 上下文工程：统筹 RAG/Prompt/Tool/Memory 动态构建上下文 | C2_3 | 中 | 理解 | 窄化为「只优化模板」或「只扩大窗口」 |

**高频考点提示**：分隔符 vs 要素辨析、系统角色作用、Few-shot/CoT 适用场景——高频。

## 模块三 · 大模型检索增强（20%，占比最高）

**教材来源**：C2_2 扩展知识范围、C2_5 优化RAG应用、C2_4 自动化评测。
**模块概述**：用 LlamaIndex 构建 RAG，掌握文件解析→切片→召回→重排核心链路；理解句子窗口检索、自动合并检索等召回优化与贴近实战的优化方法；了解 RAGAS 指标体系与 RAG 评测。

| 知识点 | 教材位置 | 权重 | 形式 | 常见易错点 |
|---|---|---|---|---|
| RAG 核心链路：文件解析→文本切片(chunk)→向量召回→重排序→生成 | C2_5 | 高 | 理解 | 颠倒环节顺序；遗漏「重排」 |
| RAG 解决「知识范围/时效/幻觉」而非提升模型推理力 | C2_2 | 高 | 理解 | 误以为 RAG 能让模型更聪明/替代微调 |
| 文本切片：chunk size / overlap 的权衡 | C2_5 | 高 | 应用 | chunk 越大越好/越小越好的绝对化 |
| 向量召回 + Top-K 检索 | C2_5 | 高 | 理解 | 把召回数量 K 越大当成一定越准 |
| 重排序（rerank）：对初召回结果二次精排 | C2_5 | 高 | 应用 | 误以为有重排就不需要好的召回 |
| 句子窗口检索（SentenceWindow）：命中句精排、返回扩展上下文 | C2_5 | 高 | 理解 | 与「自动合并检索」混淆 |
| 自动合并检索（AutoMerging，层级 node）：子块命中合并为父块 | C2_5 | 高 | 理解 | 与句子窗口混淆；误以为是简单加大 chunk |
| 实战优化：优化文本解析、标题改写、表格内容增强、分割方法对比 | C2_5 | 中 | 应用 | 忽视解析质量，直接归因于模型 |
| 嵌入模型（embedding）质量影响召回 | C2_5 | 中 | 理解 | 误以为换大语言模型即可改善召回 |
| RAGAS 指标：faithfulness 忠实度、answer_relevancy、context_precision、context_recall | C2_4 | 高 | 理解 | 混淆 precision 与 recall；忠实度与相关性混淆 |
| 忠实度低 → 答案脱离检索上下文（幻觉） | C2_4 | 中 | 应用 | 错把忠实度当成「答案是否相关」 |
| context_recall 低 → 召回不全；context_precision 低 → 召回含噪 | C2_4 | 中 | 应用 | 两者诊断方向搞反 |
| RAG 分层白盒评测：分别评检索/生成定位瓶颈 | C2_4 | 中 | 应用 | 只看端到端答案，无法定位是检索还是生成出错 |

**高频考点提示**：核心链路顺序、句子窗口 vs 自动合并、RAGAS 四指标辨析、precision/recall 诊断方向——占比最高，重点。

## 模块四 · 大模型微调（16%）

**教材来源**：C4_1 用蒸馏让小模型掌握专业能力。
**模块概述**：理解微调的作用、前提、基本步骤与常用算法（LoRA 等）；掌握微调数据集构建、关键参数与微调模型评测；理解蒸馏让小模型习得大模型能力。

| 知识点 | 教材位置 | 权重 | 形式 | 常见易错点 |
|---|---|---|---|---|
| 微调作用：让模型习得特定风格/格式/领域行为 | C4_1 | 高 | 理解 | 误以为微调能给模型「补充新知识/最新数据」（那是 RAG） |
| 微调 vs RAG vs 提示词 的选型边界 | C4_1 | 高 | 应用 | 凡是答错就微调；忽视先试 Prompt/RAG |
| 微调前提：有足够高质量标注数据、明确目标 | C4_1 | 高 | 理解 | 误以为少量脏数据也能显著见效 |
| 基本步骤：数据准备→训练→评测→部署 | C4_1 | 中 | 识记 | 跳过评测直接上线 |
| 常用算法：全参微调 vs LoRA/QLoRA（低秩、省显存） | C4_1 | 高 | 理解 | 误以为 LoRA 一定不如全参；混淆 LoRA 与量化 |
| 数据集格式：instruction/input/output 的 JSONL | C4_1 | 中 | 应用 | 字段缺失/格式不统一导致训练失败 |
| 关键参数：学习率、epoch、batch size | C4_1 | 中 | 应用 | epoch 越多越好（过拟合） |
| 过拟合与灾难性遗忘 | C4_1 | 中 | 理解 | 把训练损失下降等同于效果好 |
| 蒸馏：用大模型(教师)产出数据/软标签训练小模型(学生) | C4_1 | 高 | 理解 | 误以为蒸馏=简单压缩参数/量化 |
| 蒸馏价值：小模型逼近大模型效果，降推理成本 | C4_1 | 中 | 理解 | 误以为蒸馏后小模型必然全面超过教师 |
| 微调模型评测：用独立测试集对比基线 | C4_1 | 中 | 应用 | 用训练集评测；无基线对照 |

**高频考点提示**：微调 vs RAG 边界（微调不补知识）、LoRA 特点、蒸馏本质、过拟合/epoch——高频易错。

## 模块五 · 多Agent及多模态应用（16%）

**教材来源**：C3 全章（3_1~3_7）+ 百炼 Assistant API。
**模块概述**：理解智能体运行机制（工具调用、规划与执行、反馈），掌握多 Agent 协作模式、Memory、Skill 与评测驱动开发；基于百炼 Assistant API 生成多模态内容、构建语音助手。

| 知识点 | 教材位置 | 权重 | 形式 | 常见易错点 |
|---|---|---|---|---|
| 工具调用本质：模型决定「调什么、传什么」，执行由宿主完成 | C3_1 | 高 | 理解 | 误以为模型输出结构化对象=工具已执行成功 |
| 工具调用可靠性：schema 约束 + 程序侧参数校验 + 失败回填重试 | C3_1 | 高 | 应用 | 盲调工具不校验；只靠提示词「提醒小心」 |
| 统一工具描述协议（能力发现/参数描述/结果标准化） | C3_1 | 中 | 应用 | 为每个系统写长 Prompt 背接口；靠微调记 API |
| ReAct：推理与行动交替 | C3_1,C3_2 | 中 | 理解 | 与固定工作流混淆 |
| 固定工作流 vs Plan & Execute：步骤固定可审计→工作流 | C3_2 | 高 | 应用 | 任务稍复杂就上 Plan，忽视可审计/可控 |
| 外部反馈优于纯自我反思（错误仅在真实执行后暴露时） | C3_2 | 高 | 应用 | 让 Agent 不看报错「独立反思」；急于换大模型 |
| 多 Agent 协作：Leader-Worker（中心拆解分配汇总） | C3_3 | 高 | 理解 | 与 handoff、Blackboard 混淆 |
| handoff：顺序交接控制权、无中心调度 | C3_3 | 中 | 理解 | 误以为有最终统一输出就是 Leader-Worker |
| Blackboard：多 Agent 围绕共享中间状态并行读写迭代 | C3_3 | 中 | 理解 | 误判为固定工作流 |
| 大模型无状态 → Memory 的必要性 | C3_4 | 高 | 理解 | 误以为模型自带记忆 |
| 短期记忆（上下文）vs 长期记忆（外部存储+检索） | C3_4 | 中 | 理解 | 把所有历史无差别写入长期记忆 |
| 滚动摘要压缩长对话，保留后续决策依赖的背景 | C3_4 | 中 | 应用 | 直接截断丢早期关键信息；全量永久入库 |
| Skill：把流程固化为可复用/共享/演进的能力（触发条件+步骤+资源） | C3_5 | 高 | 应用 | 靠临时拼长 Prompt；把流程塞进会话历史 |
| 评测驱动 Agent 开发：分链路白盒评测定位瓶颈 | C3_6 | 高 | 应用 | 只扩端到端集；凭经验猜模块 |
| 完整可迭代 Agent = 工具调用+Skill+持久记忆+评测闭环 | C3_2~3_6 | 中 | 应用 | 用「更长上下文+更强模型」替代体系能力 |
| Qwen Code：CLI 编码 Agent 实践 | C3_7 | 中 | 识记 | 误以为是模型名而非工具 |
| 百炼 Assistant API：构建智能体、生成多模态内容、个性化语音助手 | 考纲/C3 | 中 | 应用 | 误以为多模态=仅文生图 |

**高频考点提示**：工具调用本质、工作流 vs Plan、三种协作模式辨析、Memory/Skill 价值、评测闭环——高频。

## 模块六 · 生产环境应用实践（16%）

**教材来源**：C4_2 部署模型、C4_3 生产实践、C4_4 安全合规。
**模块概述**：了解内容安全合规检测类型与方案、云服务环境应用系统安全要素；掌握 vLLM 部署、在 ECS/FC/PAI/百炼 部署模型，用函数计算 FC 快速发布 AI 助手。

| 知识点 | 教材位置 | 权重 | 形式 | 常见易错点 |
|---|---|---|---|---|
| 内容安全问题类型：违规生成、提示词注入、越狱、敏感信息泄露 | C4_4 | 高 | 理解 | 只防输出，忽视输入侧注入/越狱 |
| 合规检测：输入检测 + 输出检测双侧 | C4_4 | 高 | 应用 | 误以为只做输出过滤即可 |
| 内容安全方案：调用专门内容安全服务/护栏 | C4_4 | 中 | 应用 | 靠提示词「请勿生成违规内容」当主防线 |
| 云上应用安全要素：身份认证、权限最小化、密钥管理、网络隔离、审计 | C4_4 | 中 | 理解 | 把「安全」窄化为只防内容违规 |
| 知识库访问控制（按角色过滤可见文档） | C4_4 | 中 | 应用 | RAG 不做权限隔离致越权检索 |
| vLLM 部署：高吞吐推理（PagedAttention、并发） | C4_2 | 高 | 应用 | 误以为 vLLM 是训练/微调框架 |
| 部署载体选型：PAI-EAS、ECS、函数计算 FC、百炼 | C4_2 | 高 | 应用 | 不分负载特征乱选；混淆各服务定位 |
| 函数计算 FC：Serverless 快速发布 AI 助手，按量付费、弹性 | C4_2,考纲 | 高 | 应用 | 误以为 FC 适合持续高并发大模型常驻推理 |
| 量化/并发等推理性能优化手段 | C4_2 | 中 | 理解 | 把量化等同于蒸馏 |
| 生产实践：监控、可观测性、限流降级、成本控制 | C4_3 | 中 | 应用 | 上线即结束，无监控与降级预案 |

**高频考点提示**：输入+输出双侧检测、vLLM 定位（推理非训练）、FC/EAS/ECS 选型、注入与越狱——高频易错。

## 易混淆概念对照速记（命题干扰项高发区）

| A | B | 核心区别 |
|---|---|---|
| temperature | top_p | 前者整体随机性温度，后者按累计概率截断候选集 |
| 句子窗口检索 | 自动合并检索 | 前者命中句精排+扩展上下文窗口；后者层级块、子块命中合并为父块 |
| context_precision | context_recall | 精确率看召回是否含噪，召回率看是否召全 |
| 忠实度(faithfulness) | 答案相关性(answer_relevancy) | 前者答案是否忠于上下文，后者答案是否切题 |
