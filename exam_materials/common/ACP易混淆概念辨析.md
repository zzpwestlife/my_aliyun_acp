## Temperature、Top-P、Top-K 辨析

### 1. Temperature

- 中文：采样温度。
- 作用：调整模型输出的随机性和发散程度。
- 规律：越低越稳定、越保守；越高越随机、越有创意。
- 考试里怎么考：常考“它控制什么”。标准答案通常是“控制随机性”，不是控制长度，也不是直接提升质量。
- 一句话记忆：`Temperature` 调的是“敢不敢偏离最稳答案”。

### 2. Top-P

- 中文：核采样，`nucleus sampling`。
- 作用：只在“累计概率达到 `p`”的那一小批候选 token 中采样。
- 规律：`top_p` 越小，候选集越保守；越大，候选集越开放。
- 考试里怎么考：常考 `top_p=0.1` 的含义，正确理解是“只从累计概率前 10% 左右的高概率 token 集合中采样”，不是“取概率最低的 token”。
- 一句话记忆：`Top-P` 控制的是“候选池开多大”。

### 3. Top-K

- 中文：按前 `K` 个高概率 token 采样。
- 作用：只在概率最高的前 `K` 个 token 里挑选下一个 token。
- 与 `Top-P` 的区别：`Top-K` 是“固定数量”，`Top-P` 是“动态数量、看累计概率”。
- 备注：考试大纲里直接出现的是 `temperature`、`top_p`，但工程实践和面试里常把 `Top-K` 一起考，所以非常值得顺手记住。
- 一句话记忆：`Top-K` 控制的是“最多放几个候选”。

### 4. 三者关系

| 术语 | 本质 | 控制什么 | 常见误区 |
| --- | --- | --- | --- |
| `temperature` | 重塑概率分布 | 随机性强弱 | 误以为控制长度或质量 |
| `top_p` | 按累计概率截断候选集 | 候选池范围 | 误以为 `top_p=0.1` 是取低概率 token |
| `top_k` | 按候选数量截断候选集 | 候选池大小 | 误以为一定比 `top_p` 更高级 |

- 最短记忆：
  - `temperature`：改“随机程度”
  - `top_p`：改“候选池概率范围”
  - `top_k`：改“候选池数量上限”

### 5. Stream / Batch Generation

- `stream=True`：流式生成，一边生成一边返回，优点是首字延迟低、交互感更好。
- `batch/non-stream`：非流式，等完整结果生成后一次性返回，适合后台处理。
- 易混点：流式生成改善的是体验，不直接提升答案质量。

### 6. Token

- 中文：模型处理文本时的基本切分单位。
- 不是“一个汉字=一个 token”，也不是“一个单词=一个 token”，实际切分由分词器决定。
- 参数如 `temperature`、`top_p`、`top_k`，本质上都是在控制“下一步 token 怎么选”。

## RAGAS 四指标辨析

| 指标 | 主要看哪一层 | 核心问题 | 最短记忆 |
| --- | --- | --- | --- |
| `context_precision` | 检索层 | 找来的资料准不准 | 有没有找准 |
| `context_recall` | 检索层 | 找来的资料全不全 | 有没有找全 |
| `faithfulness` | 生成层 | 答案有没有忠于上下文 | 有没有胡编 |
| `answer_relevancy` | 生成层 | 答案有没有真正回答问题 | 有没有跑题 |

- 最重要的关系：
  - `context_precision` 和 `context_recall` 主要评估检索质量。
  - `faithfulness` 和 `answer_relevancy` 主要评估答案质量。
  - 前两个偏“资料找得怎么样”，后两个偏“答案答得怎么样”。

### 1. context_precision

- 含义：召回出来的上下文里，有多少是真正相关的。
- 什么时候重点看：当你怀疑检索结果混进了很多无关内容时。
- 低意味着：噪声多、召回不够准，模型容易被带偏。
- 典型场景：问“离职年假折算”，却召回了入职培训、节日福利、工资发放等无关内容。

### 2. context_recall

- 含义：回答问题需要的关键信息，有没有被充分召回。
- 什么时候重点看：当答案总是“差一点”、缺条件、缺依据时。
- 低意味着：资料没找全，关键信息缺失。
- 典型场景：问题需要 3 条制度依据，结果只召回了其中 1 条。

### 3. faithfulness

- 含义：答案是否忠于检索上下文，答案里的说法能不能在上下文中找到依据。
- 什么时候重点看：当你怀疑模型在“脑补”或“编”时。
- 低意味着：答案脱离了上下文，出现幻觉。
- 典型场景：上下文没写“必须部门审批”，答案却自己补出这个规则。

### 4. answer_relevancy

- 含义：答案是否切题，是否真正回答了用户的问题。
- 什么时候重点看：当答案看起来通顺，但没有答到问题核心时。
- 低意味着：答案跑题、绕圈、没有正面回答。
- 典型场景：用户问“怎么折算”，答案却只泛泛介绍公司休假制度。

### 最容易混的两组

- `context_precision` vs `context_recall`
  - 前者看“准不准”，后者看“全不全”。
  - 资料很多但杂，通常是 `context_precision` 低。
  - 资料不杂但缺关键点，通常是 `context_recall` 低。

- `faithfulness` vs `answer_relevancy`
  - 前者看“有没有忠于证据”，后者看“有没有答对题目”。
  - 有可能答案很切题，但夹杂自己编的内容，此时 `answer_relevancy` 高、`faithfulness` 低。
  - 也有可能答案完全依据材料，但没有回答问题重点，此时 `faithfulness` 高、`answer_relevancy` 低。

### 怎么用这四个指标排查问题

- 如果 `context_recall` 低：
  - 优先怀疑没找全，先查解析、切片、`embedding`、`Top-K`、召回策略。

- 如果 `context_precision` 低：
  - 优先怀疑噪声太多，先查 chunk 大小、检索阈值、`rerank`、标题改写、表格解析。

- 如果 `faithfulness` 低：
  - 优先怀疑模型在编，先查提示词是否要求“仅依据上下文回答”，以及上下文是否足够清晰。

- 如果 `answer_relevancy` 低：
  - 优先怀疑答案跑题，先查用户问题理解是否偏了、检索内容是否不够直接、提示词是否没有强调“直接回答问题”。

### 一句口诀

- `context_precision`：看找得准不准。
- `context_recall`：看找得全不全。
- `faithfulness`：看答得真不真。
- `answer_relevancy`：看答得对不对题。

## 训练、微调、量化、部署辨析

| 概念 | 主要目标 | 解决什么问题 | 最短记忆 |
| --- | --- | --- | --- |
| `Training` | 让模型学习能力 | 模型会不会 | 让模型学 |
| `Fine-tuning` | 让模型更适合特定任务 | 模型做事稳不稳、像不像目标风格 | 让模型更适合你的任务 |
| `Quantization` | 降低资源占用 | 显存、成本、推理效率 | 让模型更省资源 |
| `Deployment` | 让模型真正上线 | 服务运行、并发、稳定性 | 让模型跑起来 |

### 一条主线

- 先有模型，再视情况继续 `Training/Fine-tuning`。
- 上线前常会做 `Quantization` 之类的推理优化。
- 最后通过 `Deployment` 把模型提供成可调用服务。

### 最容易混的边界

- `Fine-tuning` 是训练的一种，`Quantization` 不是训练。
- `Quantization` 往往服务于部署，但不等于部署。
- `Deployment` 是工程落地阶段，不负责给模型“补知识”。
- 如果题目强调“最新知识、时效知识”，优先想到 `RAG`，不是 `Fine-tuning`。

### 什么时候优先想到谁

- 题目强调“让模型学能力、学模式”：
  - 优先想到 `Training`

- 题目强调“特定风格、格式、任务表现更稳定”：
  - 优先想到 `Fine-tuning`

- 题目强调“显存、成本、推理效率、压缩部署门槛”：
  - 优先想到 `Quantization`

- 题目强调“上线运行、并发、稳定服务、云上部署选型”：
  - 优先想到 `Deployment`

### 一句口诀

- `Training`：让模型学。
- `Fine-tuning`：让模型更适合你的任务。
- `Quantization`：让模型更省资源。
- `Deployment`：让模型真正跑起来。

## Prompt、RAG、Fine-tuning 辨析

| 概念 | 主要解决什么问题 | 什么时候优先想到 | 一句话记忆 |
| --- | --- | --- | --- |
| `Prompt` | 让模型更好理解任务和输出要求 | 任务规则清楚，但输出不稳定、格式不统一 | 改“怎么问” |
| `RAG` | 给模型补外部知识和时效知识 | 私有知识、知识量大、知识常更新 | 补“知道什么” |
| `Fine-tuning` | 让模型稳定习得任务风格和行为 | 高重复任务、固定格式、长期稳定输出 | 改“表现成什么样” |

### 最容易混的点

- `Prompt` 不是 `RAG`
  - `Prompt` 更像“告诉模型怎么做”
  - `RAG` 更像“给模型资料再让它回答”

- `RAG` 不是 `Fine-tuning`
  - `RAG` 解决知识来源问题
  - `Fine-tuning` 解决行为习惯和任务表现问题

- 如果题目强调“最新知识、公司制度、产品手册、私有文档”：
  - 优先想到 `RAG`

- 如果题目强调“固定风格、固定 JSON、长期稳定遵循格式”：
  - 优先想到 `Fine-tuning`

- 如果题目只是“回答不够稳、提示词还没认真设计”：
  - 优先先试 `Prompt`

### 一句口诀

- `Prompt`：改提问方式。
- `RAG`：补外部知识。
- `Fine-tuning`：固化行为风格。

## 蒸馏、微调、量化辨析

| 概念 | 核心思路 | 主要目的 | 易混点 |
| --- | --- | --- | --- |
| `Distillation` | 让学生模型学习教师模型输出 | 降低推理成本，同时保留能力 | 不是量化 |
| `Fine-tuning` | 用训练数据继续训练模型 | 让模型更适合任务 | 不一定有教师模型 |
| `Quantization` | 压缩参数精度 | 降显存、降成本、提推理效率 | 不是训练 |

### 怎么区分

- `蒸馏` 和 `微调`
  - 两者都可能涉及训练
  - 核心区别在于训练数据来源
  - `微调` 常用人工标注数据
  - `蒸馏` 常用教师模型生成的数据

- `蒸馏` 和 `量化`
  - `蒸馏` 是让小模型学会能力
  - `量化` 是让模型更省资源地跑

- `微调` 和 `量化`
  - `微调` 改能力和行为
  - `量化` 改资源占用和推理成本

### 教程里的高频口径

- `数据合成蒸馏 / 黑盒蒸馏`
  - 只需教师 API
  - 适合教师是商业 API、任务是结构化输出

- `知识蒸馏 KD / 白盒蒸馏`
  - 需要教师权重
  - 学的是更细的概率分布

### 一句口诀

- `蒸馏`：让小模型学大模型。
- `微调`：让模型更适配任务。
- `量化`：让模型更轻更省。

## Function Calling、ReAct、MCP 辨析

| 概念 | 主要解决什么问题 | 核心关注点 | 一句话记忆 |
| --- | --- | --- | --- |
| `Function Calling` | 让模型按结构化格式决定调用哪个工具 | 工具定义、参数 Schema、工具选择 | 解决“怎么调工具” |
| `ReAct` | 让模型在工具调用后继续思考下一步 | 思考-行动-观察循环 | 解决“调完之后怎么继续” |
| `MCP` | 让工具定义与 Agent 解耦，支持规模化发现与接入 | 标准协议、动态发现、工具管理 | 解决“工具怎么接入生态” |

### 最容易混的点

- `Function Calling` 不等于 `ReAct`
  - `Function Calling` 重点是结构化地调用一次工具
  - `ReAct` 重点是多轮“思考-行动-观察”循环

- `MCP` 不等于 `Function Calling`
  - `Function Calling` 关注一次调用时的工具描述和参数
  - `MCP` 关注工具如何被标准化发现、管理和复用

- 教程里的 USB 比喻很重要
  - `Function Calling` 更像内部调用机制
  - `MCP` 更像统一外部接口标准

### 什么时候优先想到谁

- 题目强调“用 JSON Schema 约束工具参数输出”：
  - 优先想到 `Function Calling`

- 题目强调“模型根据工具结果继续判断、继续调用工具”：
  - 优先想到 `ReAct`

- 题目强调“新增工具要少改 Agent 代码、动态发现工具、降低维护成本”：
  - 优先想到 `MCP`

### 一句口诀

- `Function Calling`：结构化调工具。
- `ReAct`：边想边调边观察。
- `MCP`：标准化接工具。

## 固定工作流、Plan & Execute 辨析

| 模式 | 特点 | 适合场景 | 风险 |
| --- | --- | --- | --- |
| 固定工作流 | 步骤预设、流程可审计 | 流程稳定、重复执行、高可控 | 遇到异常容易卡死 |
| `Plan & Execute` | 先规划再执行，步骤可动态调整 | 探索性任务、异常多、流程不稳定 | 成本更高、结果波动更大 |

### 最容易混的点

- 不是越智能越好
  - 稳定流程优先固定工作流
  - 未知任务、工具可能失效时更适合 `Plan & Execute`

- 固定工作流可以非常强
  - 适合可预测、高频、可审计任务
  - 比如审批链、固定审查流程、稳定 RAG 管线

### 一句口诀

- 稳定流程用固定工作流。
- 未知变化多用 `Plan & Execute`。

## Leader-Worker、handoff、Blackboard 辨析

| 模式 | 核心结构 | 适合什么任务 | 一句话记忆 |
| --- | --- | --- | --- |
| `Leader-Worker` | 有中心调度，拆解、分配、汇总 | 可清晰拆解的任务 | 主管带团队 |
| `handoff` | 顺序交接控制权 | 分层规划中的具体交接方式 | 接力棒 |
| `Blackboard` | 共享空间并行读写迭代 | 开放式创意协作 | 围着白板共创 |

### 最容易混的点

- `Leader-Worker` 和 `handoff`
  - `Leader-Worker` 是协作模式
  - `handoff` 更像其中一种实现机制或交接方式

- `Leader-Worker` 和 `Blackboard`
  - 前者中心化、结构清晰
  - 后者去中心化、强调共同迭代

### 什么时候优先想到谁

- 任务目标明确、步骤有依赖：
  - 优先 `Leader-Worker`

- 任务开放、需要灵感碰撞：
  - 优先 `Blackboard`

### 一句口诀

- `Leader-Worker`：有人总控。
- `handoff`：顺序交接。
- `Blackboard`：共享白板共创。

## 短期记忆、长期记忆、截断、滚动摘要 辨析

| 概念 | 解决什么问题 | 优点 | 风险 |
| --- | --- | --- | --- |
| 短期记忆 | 保持当前会话连贯性 | 实现简单 | 会受上下文窗口限制 |
| 长期记忆 | 跨会话保留关键信息 | 可持久化、可检索 | 实现更复杂 |
| 截断 | 控制上下文长度 | 简单直接 | 容易忘记早期关键点 |
| 滚动摘要 | 压缩旧对话保留重点 | 比截断更聪明 | 摘要质量会影响结果 |

### 最容易混的点

- `短期记忆` 不等于 `长期记忆`
  - 短期记忆更像当前会话缓存
  - 长期记忆更像可检索知识库

- `截断` 不等于 `滚动摘要`
  - `截断` 是直接丢
  - `滚动摘要` 是先压缩再保留

### 什么时候优先想到谁

- 简单对话、只需保最近上下文：
  - 短期记忆 + 截断

- 长任务、要保早期目标：
  - 滚动摘要

- 跨会话、长期偏好、历史经验要复用：
  - 长期记忆

### 一句口诀

- 短期记忆保当前。
- 长期记忆保未来。
- 截断是直接忘。
- 摘要是压缩记。

## Prompt、Skill 辨析

| 概念 | 本质 | 适合什么情况 | 一句话记忆 |
| --- | --- | --- | --- |
| `Prompt` | 会话级指令 | 临时任务、一次性要求 | 一次性说明书 |
| `Skill` | 文件化、可复用、可版本化的能力模块 | 重复任务、团队共享标准 | 可复用流程包 |

### 最容易混的点

- `Prompt` 只在当前对话里生效
- `Skill` 可以沉淀到文件系统，团队复用，版本管理
- `Skill` 不只是长 Prompt
  - 它通常还带触发条件、工作流、知识文件、反模式清单

### 什么时候优先想到谁

- 只是临时交代一件事：
  - 用 `Prompt`

- 每周都重复做、团队标准要统一：
  - 用 `Skill`

### 一句口诀

- `Prompt`：临时说一次。
- `Skill`：写下来反复用。

## 直接调用 API、百炼、FC、PAI-EAS、ECS、vLLM 辨析

| 方式 | 适合场景 | 优点 | 风险/限制 |
| --- | --- | --- | --- |
| 直接调用托管 API | 初期验证、低运维团队 | 快速上线、免运维 | 限流、模型选择受平台支持限制 |
| 百炼 | 阿里云生态内快速部署 | 一键部署、接入简单 | 支持模型范围有限 |
| `FC` | 低频、突发、非强实时 | Serverless、按量付费 | 冷启动、不适合持续高并发 |
| `PAI-EAS` | 稳定在线推理、要监控和弹性 | 在线服务能力强、适合实时推理 | 固定成本更高 |
| `ECS` | 高度自定义、能自己运维 | 环境完全可控 | 运维复杂 |
| `vLLM` | 推理服务框架 | 高吞吐、OpenAI 兼容接口 | 不是云服务，也不是训练框架 |

### 最容易混的点

- `vLLM` 不是部署平台
  - 它是推理框架
  - 往往跑在 `ECS`、`PAI`、本地 GPU 等环境上

- `FC` 和 `PAI-EAS`
  - `FC` 更偏低频、事件驱动、快速发布
  - `PAI-EAS` 更偏在线实时推理

- 托管 API 和自托管部署
  - 托管 API 省心
  - 自托管更可控，但运维成本高

### 一句口诀

- 初期快上用托管 API / 百炼。
- 低频突发用 `FC`。
- 稳定在线推理用 `PAI-EAS`。
- 要完全掌控环境用 `ECS`。
- `vLLM` 是推理框架，不是云服务。

## Sentence Window、Auto-Merging 辨析

| 方法 | 核心思路 | 适合什么问题 | 一句话记忆 |
| --- | --- | --- | --- |
| `Sentence Window` | 以句子命中，再扩展前后窗口 | 需要精确定位一句话附近上下文 | 命中一句，带回周边 |
| `Auto-Merging` | 多个子块命中后向上合并父块 | 需要恢复更完整结构语义 | 多个子块，合成大块 |

### 最容易混的点

- `Sentence Window` 更强调“局部命中后补上下文”
- `Auto-Merging` 更强调“层级结构里的父子块合并”
- 它们都不是简单“把 chunk 调大”

### 一句口诀

- `Sentence Window`：命中一句，补前后。
- `Auto-Merging`：子块命中，回父块。
