## 模块一　大模型应用开发

**1.**【应用开发·百炼调用·基础】使用 OpenAI 兼容方式调用百炼平台通义千问模型时，`base_url` 应设置为？

- A. `https://dashscope.aliyuncs.com/compatible-mode/v1`
- B. `https://api.openai.com/v1`
- C. `https://bailian.console.aliyun.com/v1`
- D. `https://dashscope.aliyuncs.com/v1`

**答案解析：正确答案 A**

**核心概念：** OpenAI 兼容方式调用百炼时，`base_url` 要指向兼容接口端点，而不是控制台地址或 OpenAI 官方域名。

**为什么选 A：**
- 百炼兼容端点固定为 `https://dashscope.aliyuncs.com/compatible-mode/v1`。
- 这是用 OpenAI SDK 调用百炼模型时最容易填错的参数之一。

**其他选项为什么错：**
- B 是 OpenAI 官方地址，不是阿里云百炼地址。
- C 是控制台页面地址，不是 API 端点。
- D 少了 `/compatible-mode`，是最常见误填项。

**2.**【应用开发·temperature·基础】为营销活动批量生成需要更具多样性和创意的广告文案时，`temperature` 较合理的设置是？

- A. 设为 0.0 以保证稳定
- B. 设为负值以增强随机
- C. 适当调高（如 0.8～1.0）
- D. 保持默认且不可修改

**答案解析：正确答案 C**

**核心概念：** `temperature` 主要控制输出随机性与发散程度。

**为什么选 C：**
- 创意文案需要更多样性和发散性，适合适度调高 `temperature`。

**其他选项为什么错：**
- A 太低会导致结果过于稳定、保守。
- B `temperature` 不能设为负数。
- D 参数当然可以按业务需求调整。

**3.**【应用开发·top_p·基础】当设置 `top_p=0.1` 时，模型采样时的行为是？

- A. 考虑全部候选 Token
- B. 仅从累计概率最高、占比前 10% 的少数 Token 中采样
- C. 强制输出概率最低的 Token
- D. 与 temperature 完全无关并关闭随机性

**答案解析：正确答案 B**

**核心概念：** `top_p` 是核采样，表示从累计概率达到阈值的一小部分高概率 Token 中采样。

**为什么选 B：**
- `top_p=0.1` 表示只保留累计概率最高、占比较小的一部分候选 Token。

**其他选项为什么错：**
- A 更接近 `top_p=1` 的行为。
- C 完全说反了，不会强制选最低概率 Token。
- D `top_p` 本身就与采样随机性有关。

**4.**【应用开发·无状态·中档】某助手未把上一轮对话传给模型，结果模型对用户前文提到的项目名、约束全部“失忆”。这最直接说明大模型 API 默认具有什么特性？

- A. 默认具备长期记忆，只需更长提示词激活
- B. 默认会自动缓存并恢复上一轮状态
- C. 默认把高频信息固化为长期记忆
- D. 默认无状态，多轮对话需外部显式携带上下文

**答案解析：正确答案 D**

**核心概念：** 大模型 API 默认无状态，多轮上下文需要调用方自己维护。

**为什么选 D：**
- 若不把历史对话显式传回模型，模型就不会“记得”前文信息。

**其他选项为什么错：**
- A、B、C 都把模型描述成自带长期记忆或自动缓存，这不符合 API 的默认工作方式。

**5.**【应用开发·流式·基础】要让模型一边生成一边返回内容（打字机效果），关键参数是？

- A. `n=2`
- B. `stream=True`
- C. `top_p=1`
- D. `temperature=0`

**答案解析：正确答案 B**

**核心概念：** 流式输出用于边生成边返回内容，改善交互体验。

**为什么选 B：**
- `stream=True` 会让服务端逐步返回生成片段，形成“打字机效果”。

**其他选项为什么错：**
- A 控制返回样本数量。
- C、D 与是否流式无直接关系。

**6.**【应用开发·消息结构·中档】要在多轮对话中让模型“看到”历史，正确做法是？

- A. 每次请求把历史按 `system`/`user`/`assistant` 顺序完整放入 `messages` 列表
- B. 只发送最新一句，模型会自动找回历史
- C. 把历史拼接进 `base_url` 参数
- D. 调高 `temperature` 让模型自行回忆

**答案解析：正确答案 A**

**核心概念：** 多轮对话要靠 `messages` 显式携带历史，而不是依赖模型“自己记住”。

**为什么选 A：**
- 正确做法是每轮都把历史按角色顺序放入 `messages` 中。

**其他选项为什么错：**
- B 误以为模型会自动找回历史。
- C `base_url` 是接口地址，不用于携带上下文。
- D 调温度不会让模型“回忆”历史。

**7.**【应用开发·密钥安全·中档】关于 API Key 的存储，最推荐的做法是？

- A. 写在代码注释中便于查阅
- B. 提交到公开仓库做备份
- C. 通过环境变量或独立配置文件加载，且不纳入版本库
- D. 明文硬编码到业务逻辑里

**答案解析：正确答案 C**

**核心概念：** API Key 应做最小暴露，避免硬编码和入库。

**为什么选 C：**
- 环境变量或独立配置文件是主流、安全、可运维的做法。

**其他选项为什么错：**
- A、B、D 都会造成密钥泄露风险。

**8.**【应用开发·模型选型·中档】某高频、简单的员工意图分类任务，成本敏感且要求低延迟。模型选型较合理的是？

- A. 一律使用 qwen-max 以保证质量
- B. 选用上下文窗口最大的模型
- C. 关闭全部采样参数
- D. 在满足精度前提下选用 qwen-turbo 等轻量模型以降本降延迟

**答案解析：正确答案 D**

**核心概念：** 模型选型要结合任务复杂度、成本和延迟目标，而不是一味追求“大模型”。

**为什么选 D：**
- 高频简单分类任务通常更适合轻量模型，降本降延迟更重要。

**其他选项为什么错：**
- A、B 属于过度配置。
- C 关闭采样参数不等于完成模型选型。

**9.**【应用开发·参数·拔高】某接口要求输出尽量确定、可复现（便于自动化断言校验）。最合理的参数设置是？

- A. `temperature=1` 且 `top_p=1`
- B. 把 `temperature` 设为 0（或接近 0）
- C. 把 `top_p` 调高到 0.95
- D. 增大 `n` 取多个样本

**答案解析：正确答案 B**

**核心概念：** 要求输出稳定、可复现时，应尽量降低采样随机性。

**为什么选 B：**
- `temperature=0` 或接近 0，通常能让输出更确定。

**其他选项为什么错：**
- A、C 会保留较强随机性。
- D 返回多个样本反而不利于自动化断言。

## 模块二　大模型提示词工程

**10.**【提示词·分隔符·基础】下列关于提示词分隔符的描述，正确的是？

- A. 它主要用于标识必填字段
- B. 它主要用于压缩 Token
- C. `###`、`---`、`"""` 等常用于划分提示词的不同区块
- D. 它主要用于触发工具调用

**答案解析：正确答案 C**

**核心概念：** 分隔符用于组织提示词结构，帮助模型区分不同区块。

**为什么选 C：**
- `###`、`---`、`"""` 等常用于隔开背景、要求、输入数据和输出格式。

**其他选项为什么错：**
- A 是表单概念，不是分隔符的核心作用。
- B 分隔符不会直接压缩 Token。
- D 也不是分隔符的主要用途。

**11.**【提示词·Few-shot·基础】在意图分类提示中加入若干“输入-标签”示例，这种方法称为？

- A. Few-shot（少样本提示）
- B. 全参数微调
- C. 模型蒸馏
- D. RAG 检索增强

**答案解析：正确答案 A**

**核心概念：** 给模型若干“输入-输出示例”属于 Few-shot 提示。

**为什么选 A：**
- Few-shot 的目标是通过示例引导模型模仿任务模式。

**其他选项为什么错：**
- B、C 是训练阶段技术。
- D 是检索增强，不是提示词示例学习。

**12.**【提示词·CoT·基础】思维树（ToT）、思维图（GoT）是哪种提示技术的扩展？

- A. 预训练

- D. 思维链（CoT）

**为什么选 D：**
- 它们都属于“让模型展开推理过程”的范畴，只是组织方式更复杂。

**其他选项为什么错：**
- A、B、C 分别属于训练或多模态方向，与题意不符。

**13.**【提示词·系统角色·中档】希望模型全程保持“严谨法务”口吻，且不被用户某一句话轻易带偏。最稳妥的做法是？

- A. 在每轮 user 消息末尾附一句提醒
- B. 在 system 角色消息中设定身份、风格与行为边界
- C. 仅调低 temperature 即可固定口吻
- D. 用一个 Few-shot 例子就足够

**为什么选 B：**
- system 消息是整轮对话的最高优先级约束之一，更适合定义角色与风格。

**其他选项为什么错：**
- A 容易被后续用户消息干扰。
- C 调温度不能稳定固定口吻。
- D 一个 Few-shot 样例不够稳定。

**14.**【提示词·CoT 适用·中档】某多步逻辑推理任务正确率偏低。较合理的提示策略是？

**为什么选 A：**
- 分步推理能减少逻辑跳步和条件遗漏。

**其他选项为什么错：**
- B 只强调省 Token，不利于提高推理正确率。
- C、D 都不针对推理本身。

**15.**【提示词·Meta Prompting·中档】关于 Meta Prompting，描述最准确的是？

**16.**【提示词·上下文工程·中档】“上下文工程（Context Engineering）”的核心思想是？

- A. 只优化提示词模板，不涉及其他
- B. 为模型精准构建并动态管理上下文，统筹 RAG/Prompt/Tool/Memory

**17.**【提示词·适用场景·中档】某需求要对财务明细精确到分地求和。较合理的做法是？

- A. 让模型直接“心算”求和
- B. 调高 temperature 提升计算能力

**其他选项为什么错：**

**18.**【提示词·模板·拔高】要批量审阅上千份合同，每份套用相同指令、仅替换正文。最合适的做法是？

- A. 每份都手写一遍完整提示词
- B. 把所有合同正文一次性拼进同一个提示词

**为什么选 C：**

**其他选项为什么错：**

- B 会造成上下文过长且样本互相干扰。
- D 依赖会话历史不稳定。

## 模块三　大模型检索增强

**19.**【检索增强·核心链路·基础】一个典型 RAG 应用的处理链路顺序是？

- A. 切片 → 解析 → 生成 → 召回
- B. 召回 → 重排 → 解析 → 切片
- C. 解析 → 生成 → 召回 → 重排
- D. 文件解析 → 文本切片 → 向量召回 → 重排序 → 生成

**答案解析：正确答案 D**

**核心概念：** 典型 RAG 链路是：解析 -> 切片 -> 召回 -> 重排 -> 生成。

**为什么选 D：**
- 这是最符合工程实践的标准顺序。

**其他选项为什么错：**
- A、B、C 都把步骤顺序打乱了。
- 做题时要特别注意别漏掉 `Rerank`。

**20.**【检索增强·作用·基础】引入 RAG 主要解决的问题是？

- A. 提升模型的数学推理能力
- B. 补充模型缺失/时效性知识并减少幻觉
- C. 降低推理显存占用
- D. 替代提示词工程

**答案解析：正确答案 B**

**核心概念：** RAG 的主要价值是补知识、降幻觉，而不是提升基础推理能力。

**为什么选 B：**
- 当模型知识不全、知识有时效性，RAG 可以通过外部检索补足。

**其他选项为什么错：**
- A 是推理能力问题。
- C 更偏部署工程问题。
- D RAG 与提示词工程并不是替代关系。

**21.**【检索增强·句子窗口·中档】关于句子窗口检索（Sentence Window）的特点，正确的是？

- A. 以单句为检索单元，命中后返回其前后扩展窗口作为上下文
- B. 把命中子块合并为更大的父块
- C. 直接把整篇文档作为一个 chunk
- D. 只检索标题不检索正文

**答案解析：正确答案 A**

**核心概念：** 句子窗口检索强调“句级精确命中 + 上下文窗口补充”。

**为什么选 A：**
- 它先用单句定位，再返回前后若干句作为上下文。

**其他选项为什么错：**
- B 描述的是自动合并。
- C、D 都不是句子窗口检索的特点。

**22.**【检索增强·自动合并·中档】关于自动合并检索（Auto-merging）的描述，正确的是？

- A. 命中句返回前后窗口
- B. 随机合并相邻 chunk
- C. 基于层级节点，多个子块命中时合并召回其父块
- D. 把多个文档的向量求平均

**答案解析：正确答案 C**

**核心概念：** 自动合并检索强调层级结构下“子块命中 -> 合并父块”。

**为什么选 C：**
- 当多个子节点都命中时，召回父块通常能保留更完整语义。

**其他选项为什么错：**
- A 是句子窗口的描述。
- B、D 都不符合自动合并的机制。

**23.**【检索增强·RAGAS·中档】RAGAS 的 `context_recall` 指标偏低，最可能说明？

- A. 召回的上下文不全，遗漏了回答所需信息
- B. 召回上下文中无关噪声过多
- C. 答案没有忠于检索到的上下文
- D. 答案与问题不相关

**答案解析：正确答案 A**

**核心概念：** `context_recall` 关注“该召回的信息有没有被召回到”。

**为什么选 A：**
- 该指标低通常意味着回答所需上下文缺失，属于“召回不全”。

**其他选项为什么错：**
- B 更接近 `context_precision` 低。
- C、D 更偏答案侧表现，不是召回完整性。

❗ **24.**【检索增强·忠实度·中档】RAGAS 的 `faithfulness`（忠实度）偏低，最可能说明？

- A. 答案跑题、与问题无关
- B. 召回不全
- C. 召回含噪声
- D. 答案包含了检索上下文未支持的内容（幻觉）

**答案解析：正确答案 D**

**核心概念：** `faithfulness` 衡量答案是否忠于检索上下文。

**为什么选 D：**
- 若答案里出现上下文未支持的内容，就说明忠实度低，也就是出现幻觉。

**其他选项为什么错：**
- A 更像答案相关度问题。
- B、C 属于上下文侧，不是答案是否“忠于依据”。

**易混点：**
- `context_recall` 看“找全没有”。
- `context_precision` 看“找准没有”。
- `faithfulness` 看“答的时候有没有编”。

**25.**【检索增强·重排·中档】初步向量召回的 Top-20 中，真正相关的段落排序靠后，生成时被忽略。最直接的改进是？

- A. 把 chunk 调到最大
- B. 加入重排序（rerank）对召回结果二次精排，将最相关段落提前
- C. 换更大的生成模型
- D. 关闭召回让模型直接作答

**答案解析：正确答案 B**

**核心概念：** 当初召回结果里“相关内容排得太后”时，应优先考虑 `Rerank`。

**为什么选 B：**
- 重排会对候选结果做二次精排，把最相关段落提前。

**其他选项为什么错：**
- A 调大 chunk 往往带来更多噪声。
- C 换更大的生成模型解决不了排序问题。
- D 关闭召回会失去事实依据。

**26.**【检索增强·切片·中档】chunk 过大导致召回段落混入大量无关内容、精确率下降。较合理的调整是？

- A. 继续增大 chunk
- B. 取消切片、整篇入库
- C. 减小 chunk 并设置适当 overlap，必要时配合句子窗口
- D. 仅靠提高 Top-K 解决

**答案解析：正确答案 C**

**核心概念：** chunk 过大容易引入无关内容，损害检索精度。

**为什么选 C：**
- 减小 chunk 并适当设置 overlap，可以兼顾语义完整性与检索精度。

**其他选项为什么错：**
- A 会加剧噪声问题。
- B 整篇入库失去检索粒度。
- D 单靠调 Top-K 不能从根上解决 chunk 粒度不合理的问题。

**27.**【检索增强·实战优化·拔高】文档含大量表格，RAG 对表格数据问答总是答错。最对症的优化是？

- A. 优化解析与表格内容增强（表格结构化、补充表头语义）
- B. 直接调高 temperature
- C. 增大 Top-K 召回更多文本
- D. 仅换用更高维度的 embedding

**答案解析：正确答案 A**

**核心概念：** 表格类问答失败，常见根因在于文档解析与结构保留，而不只是模型能力不够。

**为什么选 A：**
- 表格需要保留表头、单元格对应关系和结构语义，解析质量至关重要。

**其他选项为什么错：**
- B 调温度属于生成参数。
- C 增大 Top-K 可能只会召回更多噪声。
- D 只换 embedding 不能替代表格结构解析。

**28.**【检索增强·分层评测·拔高】RAG 系统答案质量下降，难以判断是检索还是生成出了问题。优先应该？

- A. 直接同时更换检索与生成模块
- B. 扩大端到端评测集后再慢慢猜
- C. 凭经验先改最可能的模块
- D. 对检索与生成分别做白盒评测，先定位是哪一层退化

**答案解析：正确答案 D**

**核心概念：** RAG 退化时，先定位是检索层还是生成层出问题，比直接乱改更重要。

**为什么选 D：**
- 白盒分层评测能先定位瓶颈，避免同时改多个模块导致难以归因。

**其他选项为什么错：**
- A、B、C 都缺乏定位思维，不利于高效排障。

## 模块四　大模型微调

**29.**【微调·选型边界·基础】希望模型能回答“公司最新产品手册”的内容。最合适的做法是？

**核心概念：** 微调不擅长补最新事实知识，更新频繁的知识更适合 RAG。

**为什么选 B：**

**30.**【微调·作用·基础】模型微调最擅长的是？

- 当目标是稳定输出某种风格或任务模式时，微调更对路。

- A 是 RAG 更擅长的事情。

**31.**【微调·LoRA·中档】关于 LoRA，描述正确的是？

- C. 效果必然差于全参数微调
- D. 主要用于数据增强

**核心概念：** LoRA 是参数高效微调方法，只训练少量附加参数。

**为什么选 A：**
- 它通过低秩适配降低训练成本和显存消耗。

**其他选项为什么错：**
- B 把 LoRA 和量化混淆。
- C 说得过于绝对。

**32.**【微调·过拟合·中档】微调时训练损失持续下降，但测试集表现反而变差。最可能的原因是？

**其他选项为什么错：**
- A、B、D 都不能直接解释“训练好、测试坏”的模式。

**33.**【微调·蒸馏·中档】模型蒸馏（distillation）的本质是？

- A. 把模型权重量化到低比特
- B. 用大模型（教师）产出的数据/软标签训练小模型（学生）以迁移能力

- D. 把多个模型集成投票

**答案解析：正确答案 B**

**核心概念：** 蒸馏是“教师模型教学生模型”，核心是能力迁移。

**为什么选 B：**

**其他选项为什么错：**
- A 是量化。
- C 是剪枝。
- D 是集成。

**易混点：**
- 蒸馏：大模型教小模型。
- 量化：把权重压到低比特。
- 剪枝：删掉不重要结构。

**34.**【微调·前提·中档】团队想微调，但目前只有几十条质量参差的样本。最该先做的是？

- 只有几十条且质量参差的样本，先扩充和清洗数据才是正确顺序。

**其他选项为什么错：**

**35.**【微调·数据格式·中档】监督微调（SFT）数据集常见的组织形式是？

**36.**【微调·评测·拔高】要证明微调后的模型确实变好，最严谨的做法是？

**39.**【多Agent·外部反馈·中档】代码 Agent 在本地静态检查全过，但集成环境报错。想快速分辨是“代码逻辑”还是“运行条件”问题，最有效的下一步是？

**40.**【多Agent·协作模式·中档】一个调度 Agent 接收任务、拆分子任务并分给检索/写作/审校 Agent，最后统一汇总。这最接近哪种协作模式？

**42.**【多Agent·记忆·中档】超长投诉会话中，早期消息含用户身份、历史订单和已承诺补偿，上下文将满。既不想全丢早期信息，也不想原文无差别入库。较合适的是？

**43.**【多Agent·Skill·中档】代码审查每次临时拼长 Prompt、成员写法不一、结果不稳定。若想沉淀可复用、可共享、可演进的能力，最适合？

**71.**【生产实践·安全风险·基础】下列属于大模型应用内容安全风险的有？

**72.**【生产实践·合规手段·中档】内容安全合规的合理手段包括？

**75.**【生产实践·生产实践·拔高】大模型应用的生产实践应考虑哪些方面？
